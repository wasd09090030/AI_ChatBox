# plan_2026-03-17_故事调整功能实施方案

## 1. 背景与目标

本方案针对“故事设计”侧的新增需求，设计一个独立的“故事调整”功能页，用于对已经生成并持久化的故事内容进行局部润色与保存。

目标如下：

1. 在左侧导航“功能”区新增路由“故事调整”。
2. 在该页面中按当前故事 `segments` 顺序展示所有已由大模型输出的内容。
3. 用户拖动鼠标选中文本后，出现“AI 润色”入口。
4. 润色调用继续沿用当前故事创作使用的模型配置链路，但不能污染正式故事续写上下文。
5. 润色 prompt 支持“预设方案 + 用户自定义指令”组合。
6. 调整页中的修改必须支持“撤销未保存改动”和“保存全部改动”。
7. 保存后要同步更新数据库中的故事内容，并让故事创作页显示最新文本。
8. 保存后还要处理与 `story session`、`summary memory`、`conversation history` 的一致性。

本方案额外确认的产品决策：

- 预设润色方案先写死在前端常量里，不做后台可配置化。

## 2. 现状基线与代码事实

以下事实来自当前仓库代码，而不是假设。

### 2.1 前端现状

- 当前主故事页路由只有 `/story`，见 `frontend/src/router/index.ts`。
- 左侧“功能”区不是动态读取路由，而是在 `frontend/src/components/layout/AppSidebar.vue` 中硬编码 `featureItems`。
- 当前故事创作页是 `frontend/src/views/StoryView.vue`。
- 故事列表、当前故事、选中世界、`v2SessionId` 目前由 `frontend/src/domains/story/composables/useStoryLibrary.ts` 在页面局部维护。
- 当前故事结构来自 `frontend/src/components/story/types.ts`：
  - `StoredStory.segments` 为顺序数组。
  - `StorySegment` 只有 `id / prompt / content / choices / retrieved_context / timestamp`。
- 当前前端只支持“新增 segment”，不支持“修改已有 segment”，见 `frontend/src/domains/story/api/storyLibraryApi.ts`。
- 当前故事创作页在追加段落后，会直接把新的 `segment` push 到当前故事对象里，再调用后端持久化，见 `frontend/src/domains/story/composables/useStoryGeneration.ts`。
- 应用层已启用 `KeepAlive`，见 `frontend/src/App.vue`。这意味着故事创作页在路由切换后可能仍保留内存态。

### 2.2 后端现状

- 故事 CRUD 与 segment 追加接口位于 `story_rag_service/api/v2/world_story_routes.py`。
- `StoryManager` 目前支持：
  - `create_story`
  - `update_story`
  - `add_segment`
  - `update_story_progress`
  - 不支持更新已有 segment 内容，见 `story_rag_service/services/story_manager.py`。
- 故事持久化模型 `StoredStory` 位于 `story_rag_service/models/stored_story.py`。
- 当前 `stories` 运行时真相源是 SQLite 中 `stories` 表，`payload` 字段存整条故事 JSON，见 `story_rag_service/repositories/story_repository.py`。

### 2.3 生成与记忆链路现状

- 当前非流式故事生成入口是 `POST /api/v2/story/generate`，见 `story_rag_service/api/v2/story/generation_routes.py`。
- 该链路最终调用 `StoryGenerator.generate_story(...)`，见 `story_rag_service/services/story_generator.py`。
- 当前生成不仅返回文本，还会在生成后执行：
  - 向 `story_session_messages` 写入 user/assistant turn
  - 更新 session cache
  - 更新或异步刷新 summary memory
  - 将超出窗口的历史写入长期记忆向量索引
- 以上逻辑由 `MemoryUpdateService.run_post_generation_updates(...)` 触发，见 `story_rag_service/application/memory/update_service.py`。
- `story session` 的数据库访问由 `story_rag_service/repositories/story_session_repository.py` 负责。
- `summary memory` 的真相源是 `conversation_summaries` 表，见 `story_rag_service/services/summary_memory_manager.py`。
- 长期历史记忆通过 `ConversationHistoryManager.add_message_to_rag(...)` 写入向量索引，见 `story_rag_service/services/conversation_history_manager.py`。

### 2.4 从现状推导出的关键结论

1. 不能直接把“润色操作”复用成一次普通 `/story/generate` 调用，否则会把一次编辑动作写入正式故事会话。
2. 不能只更新 `stories` 表里的 `segments.content`，否则后续续写仍会读取旧的 `story_session_messages`、旧 summary、旧历史向量记忆。
3. 要想让“故事调整”真的可用，必须把“故事文本保存”和“记忆链路对齐”一起设计。

## 3. 范围、约束与非目标

### 3.1 本期范围

- 新增故事调整路由与页面。
- 复用现有世界/故事选择方式。
- 支持在单个 AI 段落内选中文本并触发 AI 润色。
- 支持前端预设润色方案与自定义 prompt。
- 支持未保存草稿撤销。
- 支持保存全部改动到数据库。
- 保存后同步更新故事创作页与后端记忆链路。

### 3.2 明确约束

- 预设润色方案先写死在前端常量里。
- 继续沿用当前 provider/model 配置体系。
- 不能破坏现有故事创作流程与路由结构。
- 不能让调整操作污染正式续写上下文。

### 3.3 本期非目标

- 不支持跨多个 `segment` 的选区润色。
- 不支持保存后的持久版本回溯。
- 不做可配置的后台润色模板管理。
- 不改造现有段落模型为字符级 patch 存储。

## 4. 核心设计决策

### 4.1 路由与页面决策

- 新增功能页路由：`/story-adjustment`
- 页面名称：`StoryAdjustmentView.vue`
- 左侧导航“功能”区新增“故事调整”

选择原因：

- 与当前 `/story` 同级，符合“功能区”分类。
- 不把调整逻辑硬塞进 `StoryView.vue`，避免故事创作页继续膨胀。
- 与现有 `KeepAlive` 策略兼容，更利于两页共享当前故事上下文。

### 4.2 状态管理决策

- 将当前 `useStoryLibrary.ts` 中的长期工作区状态抽离为共享 store。
- 推荐新增 `frontend/src/stores/storyWorkspace.ts`。

共享内容建议包含：

- `selectedWorldId`
- `stories`
- `storiesLoading`
- `currentStory`
- `v2SessionId`
- `loadStories`
- `createStory`
- `deleteStory`
- `selectStory`
- `ensureV2Session`
- `syncStoryLocally`

选择原因：

- 当前这些状态只存在于故事创作页局部，故事调整页无法直接复用。
- 若不共享状态，保存后 StoryView 与 StoryAdjustmentView 会各自持有不同版本的 `currentStory`。

### 4.3 润色调用决策

- 不直接调用当前 `/api/v2/story/generate`。
- 新增无副作用润色接口，例如：
  - `POST /api/v2/story/adjustments/polish`

这个接口的“复用同一 API 调用”应解释为：

- 复用相同 provider/model/base_url/temperature 的模型配置链路。
- 复用相同 `X-User-ID` 头与 provider 选择规则。
- 复用相同 LLM 创建工厂。
- 不复用当前会写 session/messages/summary/history 的生成副作用。

### 4.4 保存接口决策

- 新增批量提交接口，例如：
  - `POST /api/v2/stories/{story_id}/adjustments/commit`

提交内容按“完整 segment 文本”提交，而不是字符级 patch。

选择原因：

- 当前 `stories.payload` 已经是整故事 JSON，批量替换 segment 内容最贴近现有模型。
- 字符级 patch 需要引入 offset 迁移、服务器端 patch 校验和冲突处理，首版复杂度过高。

### 4.5 一致性决策

- 保存时必须统一执行“故事文本提交 + 会话重建 + 记忆对齐”。
- 稳定 session id 继续使用当前前端规则：`story-{story_id}-v2`。

本期一致性策略定义为：

1. `stories` 是故事正文真相源。
2. `story_session_messages` 是正式续写上下文真相源。
3. `conversation_summaries` 与 conversation history 向量索引是派生记忆。
4. 保存后必须让 2、3 对齐 1。

## 5. 目标架构

```mermaid
flowchart TD
    A[StoryAdjustmentView] --> B[选中文本]
    B --> C[预设方案 + 自定义指令]
    C --> D[POST /api/v2/story/adjustments/polish]
    D --> E[返回 replacement text]
    E --> F[仅更新前端 draftSegments]
    F --> G[用户继续调整 / 撤销]
    G --> H[点击保存]
    H --> I[POST /api/v2/stories/{story_id}/adjustments/commit]
    I --> J[更新 stories payload]
    J --> K[重建 story_session_messages]
    K --> L[重置或重建 summary memory]
    L --> M[清理并重建 session history vector index]
    M --> N[返回最新 StoredStory]
    N --> O[共享 storyWorkspace 同步]
    O --> P[StoryView 与 StoryAdjustmentView 同步显示]
```

## 6.7 润色交互流程

1. 用户在某个 AI segment 中选中文本。
2. 页面识别选区范围与所属 `segmentId`。
3. 弹出“AI 润色”入口。
4. 用户选择预设方案，可填写自定义指令。
5. 前端调用 `storyAdjustmentApi.polish(...)`。
6. 后端返回 `polished_text`。
7. 前端将选中片段替换为 `polished_text`，只更新草稿层。
8. 记录操作到 `undoStack`。

## 6.8 保存与撤销语义

### 撤销

- 仅撤销未保存草稿。
- 每次撤销回退最近一条 `StoryAdjustmentOperation`。
- 撤销后若草稿内容与原文一致，则从 `pendingChanges` 移除该 segment。

### 保存

- 点击保存时，一次性提交所有变化过的 segment：

```ts
{
  session_id: string
  updates: [
    { segment_id: string, content: string }
  ]
}
```

- 保存成功后：
  - 用后端返回的完整 `StoredStory` 替换共享工作区中的 `currentStory`
  - 清空 `draftSegments`
  - 清空 `undoStack`
  - 清空 `isDirty`

## 6.9 跨页面同步策略

同步来源必须统一为后端提交成功后返回的最新 `StoredStory`。

不要在前端自行拼装最终故事对象作为“保存成功后的真相源”。

原因：

- 后端提交后还会刷新 `updated_at`
- 后端可能附带修正元数据
- 统一使用服务端返回值可降低双端不一致

同步方式：

1. 保存成功
2. `storyWorkspace.syncStory(updatedStory)`
3. `StoryView` 由于读取同一 store，自动看到最新内容
4. 若有必要，可在 `StoryAdjustmentView` 保存成功后主动清理本地 summary 缓存

## 6.10 离开保护

若 `isDirty === true`：

- 路由切换前弹出确认
- 故事切换前弹出确认
- 刷新页面前弹出浏览器原生提示

```python
class StorySegmentContentUpdate(BaseModel):
```

### 响应体建议

```python
class StoryAdjustmentCommitResponse(BaseModel):
```

## 7.4 StoryManager 扩展

新增方法建议：

```python
def update_story_segments_content(
    self,
    story_id: str,
    updates: list[StorySegmentContentUpdate],
) -> Optional[StoredStory]:
    ...
```

实现规则：

1. 读取目标 story
2. 建立 `segment_id -> content` 映射
3. 逐个替换命中的 `segment.content`
4. 未命中的 `segment_id` 返回 400 或 404
5. 刷新 `updated_at`
6. 保存并返回完整 `StoredStory`

首版不修改 `prompt`、`choices`、`retrieved_context`、`timestamp`。

## 8. 保存后的记忆一致性方案

这是本方案最关键的部分。

## 8.1 为什么必须做一致性重建

当前继续故事不是只读 `stories`：

- 会读 `story_session_messages`
- 会读 `SessionManager` cache
- 会读 `conversation_summaries`
- 会读长期历史向量检索结果

若保存后只改 `stories`，则会出现：

- 故事创作页显示已是新文本
- 模型续写却仍按旧文本记忆继续

这会直接导致产品层面的“看起来已保存，实际上没生效”。

## 8.2 重建触发时机

- 仅在“保存”成功后触发
- 润色预览与未保存草稿阶段不触发

## 8.3 稳定 session id

继续采用当前前端规则：

`story-{story_id}-v2`

提交接口若未显式传 `session_id`，后端可按该规则推导。

## 8.4 重建步骤

建议新增 `StoryConsistencyRebuildService`，统一执行以下动作。

### 第一步：重写 session messages

根据 `StoredStory.segments` 顺序重建：

- user message = `segment.prompt`
- assistant message = `segment.content`

执行动作：

1. 清理该 session 的旧 `story_session_messages`
2. 重新写入当前故事对应的 turn 序列
3. 修正 `story_sessions.last_active_at`
4. 若存在至少一轮对话，则将 `first_message_sent` 置为真

### 第二步：刷新 SessionManager cache

当前 `SessionManager` 有内存 LRU cache。

保存后应：

1. 删除该 session 的旧 cache
2. 用重建后的 `Message[]` 重新构造 `StoryContext`
3. 重新放回 cache

### 第三步：处理 summary memory

首版建议采取“清空旧摘要，再按需重建”的务实策略：

1. 删除 `conversation_summaries` 中该 session 的旧摘要
2. 不在提交接口里同步跑一次长摘要生成
3. 后续故事继续生成时，由现有 summary 更新机制自然重建

选择这个策略的原因：

- 旧摘要不能继续保留，否则会携带被用户已经改掉的事实
- 同步重跑摘要会显著增加保存耗时
- 首版先保证“不错误”，再优化“立即可用摘要”

可选增强版本：

- 在提交成功后异步发起 summary rebuild
- 完成后刷新前端 `storySessionStore.summaryMap`

### 第四步：清理并重建长期历史向量索引

当前历史写入向量索引使用固定 id 规则：

`history_{session_id}_{turn_number}`

这为定向清理创造了条件。

建议扩展：

- `VectorStoreManager.delete_entries_by_prefix(prefix: str)`

或在 `ConversationHistoryManager` 中封装：

- `rebuild_session_history(session_id, world_id, messages)`

流程：

1. 删除前缀为 `history_{session_id}_` 的旧索引项
2. 按新的 turn 序列重新调用 `add_message_to_rag(...)`

说明：

- 长期历史是派生索引，不是主真相源
- 若这一步失败，故事正文与 session DB 仍应保留提交成功结果
- 接口响应中应通过 `warnings` 明确告知是否存在“索引重建失败但正文已保存”的情况

## 8.5 一致性优先级

提交链路中各层优先级如下：

1. `stories` 正文落库
2. `story_session_messages` 重建
3. `SessionManager` cache 刷新
4. `summary memory` 清空
5. 长期 history 向量索引重建

其中：

- 1 到 4 属于必须完成项
- 5 是强建议项，但如果索引层短暂失败，不应回滚已经成功的正文保存

## 11. 风险与缓解

### 风险 1：跨 segment 选区导致 offset 复杂度陡增

影响：

- 替换范围难以稳定映射
- 保存粒度不再清晰

缓解：

- 首版只支持单 segment 内选区

### 风险 2：调整后故事正文与续写上下文不一致

影响：

- 用户看到新文本，但模型按旧上下文续写

缓解：

- 提交接口强制触发 session/messages/summary/history 重建

### 风险 3：向量索引重建失败

影响：

- 长期历史召回可能短时间不完整

缓解：

- 正文与 session DB 提交优先
- 接口返回 `warnings`
- 日志中记录可重试信息

### 风险 4：KeepAlive 下 StoryView 持有旧引用

影响：

- 调整页已保存，但 StoryView 不立即刷新

缓解：

- 统一以共享 store 为真相源
- 保存成功后用完整 `StoredStory` 覆盖共享状态，而不是局部 mutate 草稿对象

### 风险 5：用户误以为“撤销”会回滚数据库

影响：

- 心智模型混乱

缓解：

- 按钮文案明确区分：
  - `撤销本地改动`
  - `保存到故事`

## 12. 手工验证清单

### 12.1 路由与基础展示

- 左侧功能区出现“故事调整”
- 进入后可复用当前世界与故事选择
- 切回故事创作页后当前故事不丢失

### 12.2 选区润色

- 在单个 AI 段落内选中一段文本后出现“AI 润色”
- 跨段选中时给出限制提示
- 预设方案可选，自定义 prompt 可输入
- 润色结果只作用于选中范围，不扩散到未选中文本

### 12.3 草稿与撤销

- 连续润色两次后可逐步撤销
- 未保存离开页面有提示
- 切换故事时如有脏数据有提示

### 12.4 保存与同步

- 保存后数据库中的 story payload 已更新
- 返回 StoryView 后同一段文本已是新内容
- 继续生成时，模型依据新文本推进，而不是沿用旧文本

### 12.5 失败路径

- 润色接口失败时草稿不被污染
- 保存失败时本地草稿保留
- 若 history 索引重建失败，正文仍保存成功且界面有明确提示

## 13. 最终建议

本功能的正确实现关键不在“选区润色 UI”，而在“保存后上下文是否真的切换为新文本”。

因此本方案建议按以下优先级实施：

1. 先做共享工作区与路由骨架。
2. 再做前端草稿交互与前端常量预设。
3. 再做无副作用润色接口。
4. 再做批量提交。
5. 最后完成会话与记忆重建。

若时间受限，不建议砍掉“记忆重建”，而应砍掉“跨段选区”或“保存后即时 summary 重建”。

因为前者是体验增强，后者才是正确性底线。
