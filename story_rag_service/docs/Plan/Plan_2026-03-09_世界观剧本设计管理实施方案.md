# 简化版剧本设计补充方案（2026-03-09）

## 目标

- 现有剧本设计字段较全，更适合熟悉系统的重度用户。
- 对一般用户，默认提供“简化版剧本设计”，只暴露完成主线规划所需的最小字段集。
- 基础设置始终可见；主线结构改为独立设计器模态框；进阶字段只在各自语义范围内按需展开，不改后端模型，也不拆分数据源。

## 交互原则

- 默认只展示基础设置，降低首次使用门槛。
- 基础设置按用户任务流组织，而不是按后端字段组织。
- 进阶设置不是整页切换，而是就近附着在对应内容附近的可选增强块，用于补充细粒度约束、依赖关系和策略参数。
- 基础设置与高级项编辑同一份 ScriptDesign 数据；未展开的高级字段必须被保留，不能丢失。
- 结构相关的基础与进阶编辑统一收拢到一个模态框中，避免“在这里勾选、在别处编辑”的割裂体验。

## 基础设置保留内容

### 1. 剧本概览

- 标题
- 一句话梗概
- 故事概述
- 主题
- 核心冲突
- 结局方向
- 生成提示

### 2. 主线结构

- 阶段标题
- 阶段目标
- 关键转折
- 每阶段关键事件
	- 事件标题
	- 事件概述
	- 事件目标
	- 主要阻碍
	- 预期结果

### 3. 重要伏笔

- 名称
- 伏笔内容
- 埋设阶段
- 回收阶段
- 回收说明
- 状态

## 高级设置保留内容

- 剧本状态、标签、主角定位、叙事风格
- 默认生成策略开关
- 阶段进入/退出条件、张力、关联角色、关联设定、备注
- 事件类型、状态、触发条件、失败结果、场景提示、前后置依赖、参与对象、关联伏笔
- 伏笔分类、事件级挂载、重要性、备注

## 前端实现策略

- 首页采用低密度双栏工作台：左侧是剧本信息，右侧是主线结构摘要与关联故事。
- 主线结构不再直接在首页整页编辑，而是通过“结构设计器”模态框统一管理。
- 剧本信息卡片内部使用 details/summary 形式的“进阶补充”手风琴，展开后编辑人物定位、叙事风格、状态标签与生成策略。
- 结构设计器中默认显示阶段、关键事件、重要伏笔的基础编辑；每个阶段、事件、伏笔条目内部各自提供“高级设置”手风琴，而不是使用外置开关或独立高级卡片。
- 所有进阶块与基础块共用同一份提交载荷，保存时直接提交完整结构数据，保证编辑位置与语义位置一致，同时避免未展开字段丢失。

## 收益

- 一般用户可快速完成“搭主线骨架”的最小闭环。
- 高级用户仍保留完整控制能力。
- 不增加后端复杂度，不需要迁移数据，也不会引入双模型维护成本。

# Plan_2026-03-09_世界观剧本设计管理实施方案

## 1. 目标与边界

### 1.1 目标

在现有“世界书 / Lorebook / 故事创作”体系之间新增一层“剧本设计”能力，用于为某个世界预先定义可执行的故事规划，并在后续故事创作时提供结构化引导。

该层需要解决三类问题：

1. 主线分阶段：允许将一个世界下的长期故事规划拆成多个阶段或章节弧线。
2. 特定事件细化：允许把某阶段中的关键事件拆成具体节点、前置条件、目标、参与角色与结果。
3. 伏笔记录与回收：允许记录伏笔、埋设位置、预期回收节点、当前状态，避免生成中遗忘。

### 1.2 不做的事

首版不直接替代实际“已生成故事”内容，不把剧本设计本体塞进 World 基础字段，也不把它当成普通 Lorebook 条目。

首版不做复杂图形化时间线编辑器，不做多人协作，不做自动冲突求解，只做足够稳定的结构化管理与后续生成接入。

### 1.3 最终定位

剧本设计是独立聚合，挂在某个 world_id 下，与以下对象形成并列关系：

1. World：世界设定与基础元信息。
2. Lorebook：静态设定资料、关键角色、地点、事件背景。
3. ScriptDesign：故事规划蓝图。
4. StoredStory：某次实际创作出来的故事实例。

关系原则：

1. 一个世界可有多个剧本设计。
2. 一个剧本设计可被多个故事实例引用。
3. 一个故事实例在首版最多绑定一个 script_design_id。

## 2. 推荐信息架构

### 2.1 管理入口

推荐新增独立管理入口，而不是继续把所有能力都压进现有世界书页签。

推荐路径：

1. 前端新增 Dashboard 页面入口：剧本设计。
2. 路由建议：/dashboard/scripts。
3. 页面内部以 world 为主筛选条件。

原因：

1. Lorebook 当前已经承担世界、关键角色、普通角色、地点、事件、Persona 管理，继续叠加会过重。
2. 剧本设计是“动态规划层”，语义上与静态世界书不同。
3. 后续若扩展进度跟踪、故事挂载、节点推进，这个页面更容易演进。

### 2.2 页面结构

页面建议拆成四栏或四区块逻辑，但首版可采用两栏布局：

1. 左侧：世界选择 + 剧本设计列表。
2. 主区域上半：剧本概览与主线阶段。
3. 主区域下半左：事件节点列表。
4. 主区域下半右：伏笔记录与回收状态。

首版 UI 模块建议：

1. 剧本设计列表
2. 剧本设计基础信息面板
3. 主线阶段编辑器
4. 事件节点编辑器
5. 伏笔记录编辑器
6. 关联故事面板

## 4. 与现有领域的关系设计

### 4.1 与 World 的关系

World 保持轻量，不新增剧本正文级大字段。

只需要保证 ScriptDesign 通过 world_id 归属世界即可。

### 4.2 与 Lorebook 的关系

ScriptDesign 不复制世界书正文，而是通过 id 关联：

1. linked_role_ids 关联关键角色。
2. linked_lorebook_entry_ids 关联地点、事件、设定条目。

这样可以避免世界设定和剧本规划双份维护。

### 4.3 与 StoredStory 的关系

建议在 StoredStory 上新增最小关联字段，而不是把完整剧本设计快照塞进去。

建议新增字段：

1. script_design_id: string | null
2. active_stage_id: string | null
3. active_event_id: string | null

首版目的：

1. 让某个实际故事知道自己参考的是哪个剧本设计。
2. 允许后续在故事库里筛选“来自哪个剧本设计”。
3. 为未来“继续推进下一个事件”提供挂点。

不建议首版在 StoredStory 中存完整剧本副本，否则后续世界书与剧本设计变更会造成多份数据漂移。

## 7. 分阶段开发计划

### Phase 1：建模与管理端落地

目标：先把“剧本设计”作为可保存、可编辑、可查看的管理对象做出来。

范围：

1. 后端模型、仓储、应用服务、API。
2. 前端管理页与基础 CRUD。
3. 与 world 关联、与 lorebook 只做 id 级关联。

交付标准：

1. 可以在某个世界下新增多个剧本设计。
2. 可以维护阶段、事件、伏笔。
3. 可以查看某个剧本设计关联了哪些故事。
4. 具备基础 smoke API 验证。

### Phase 2：故事创作接入

目标：让 StoryView 能显式选择并引用某个剧本设计。

范围：

1. Story 生成请求增加 script_design 相关字段。
2. Prompt 中注入当前阶段 / 当前事件 / 伏笔约束。
3. 前端故事创作控制区新增剧本设计选择。

交付标准：

1. 请求 payload 能携带 script_design_id。
2. 后端 debug 信息能看出剧本设计已生效。
3. 生成内容能体现当前阶段与事件目标。

### Phase 3：进度推进与回写

目标：让故事实例和剧本设计之间形成最小闭环。

范围：

1. StoredStory 挂载 active_stage_id / active_event_id。
2. 提供手动推进事件状态的接口。
3. 故事详情里可回看绑定的剧本设计。

交付标准：

1. 某个故事可以标记当前推进到哪个阶段 / 事件。
2. 剧本设计列表可看到已绑定故事数。
3. 伏笔状态可被人工更新。

## 9. 验证方案

### 9.1 后端 smoke 场景

建议新增脚本：

1. 新建 world
2. 新建 script design
3. 更新阶段、事件、伏笔
4. 读取列表与详情
5. 校验错误引用被正确拒绝

### 9.2 前端人工验证

必须走通的操作：

1. 在管理页创建剧本设计。
2. 新增两个阶段。
3. 在第二阶段下新增关键事件。
4. 为某事件挂一个待回收伏笔。
5. 在故事创作页选择该剧本设计并发起生成。
6. 验证请求成功、页面状态正确、生成结果未明显偏离当前事件目标。

## 10. 风险与控制

### 10.1 风险一：结构过重，难以首版落地

控制方式：

1. 首版只做单层阶段、单层事件，不做嵌套事件树。
2. 首版不做拖拽排序，先用显式 order。

### 10.2 风险二：与现有 Lorebook 语义重叠

控制方式：

1. Lorebook 只存静态设定。
2. ScriptDesign 只存故事推进规划。
3. 通过关联 id 避免复制内容。

### 10.3 风险三：故事生成接入后 prompt 过长

控制方式：

1. 只注入当前阶段、当前事件、未完成高优先级伏笔。
2. 不把整个剧本设计全文拼进 prompt。

## 11. 推荐实施顺序

建议按下面顺序开工：

1. 先完成后端 ScriptDesign 模型与 repository。
2. 再完成 application service 与 API。
3. 再完成前端管理页基础 CRUD。
4. 然后再把 StoryView 的剧本设计选择接进去。
5. 最后补 smoke 脚本、人工验证与文档更新。

## 12. 结论

“剧本设计”不应继续塞进现有世界字段或普通 Lorebook 条目，而应作为 world 之下的独立规划层落地。

从当前仓库结构看，这一方案与现有 world repository、story repository、dashboard 管理页和 story generation 显式传参模式是兼容的，实施成本可控，且能自然支持后续“故事推进状态”“伏笔回收管理”“按剧本继续生成”等扩展能力。
