# Plan_2026-03-27_记忆更新事件契约与一致性可视化实施方案

## 1. 背景与目标

当前仓库的故事记忆系统已经具备“统一读路径 + 统一写路径”的雏形，但还没有把“记忆发生了什么变化”变成一个稳定、可观测、可前端展示的产品能力。

本次规划聚焦三个直接目标：

1. 定义 `memory_updates` 事件契约。
2. 给 `rollback / regenerate` 增加 `summary / vector` 一致性处理。
3. 把前端从“快照显示”升级到“事件 + diff 可视化”。

本方案的预期产出不是新框架，而是在现有记忆架构上补齐可靠性和可视化的缺口。

---

## 2. 现状基线与代码事实

以下结论全部来自当前仓库代码与已有验证文档，而不是假设。

### 2.1 后端读路径已经收敛

- `story_rag_service/application/memory/orchestrator.py`
  - 已实现 `MemoryOrchestrator.build_bundle(...)`。
  - 已统一聚合：
    - episodic recent window
    - episodic recalled history
    - semantic summary memory
    - profile snapshot
    - procedural context
    - world / lore context
- `story_rag_service/services/story_generation/prompt_builder.py`
  - 已支持以 `MemoryBundle` 为主要输入构建系统 prompt。

这说明 `Plan_2026-03-15_记忆架构重构与迁移方案.md` 中“统一读路径”的目标已经有落地基础。

### 2.2 后端写路径已经收敛

- `story_rag_service/application/memory/update_service.py`
  - 已统一处理：
    - `story_session_messages` 写入
    - episodic 历史索引更新
    - semantic summary 更新
- `story_rag_service/services/story_generator.py`
  - 非流式、同步、流式三条生成路径都已接入 `MemoryUpdateService.run_post_generation_updates(...)`。

这说明写路径不是分散在多个 route 或 helper 里，而是已经有统一入口。

### 2.3 真相源与派生索引边界已基本正确

- `story_rag_service/repositories/story_session_repository.py`
  - `story_sessions` / `story_session_messages` 是会话与原始消息的持久化真相源。
- `story_rag_service/services/summary_memory_manager.py`
  - `conversation_summaries` 是 semantic memory 的真相源。
- `story_rag_service/services/conversation_history_manager.py`
  - 负责把历史会话投影到向量索引，并负责历史召回。
- `story_rag_service/services/vector_store/manager.py`
  - 向量库已经被明确定位为检索索引，而不是原始会话真相源。

### 2.4 已存在部分一致性重建机制

- `story_rag_service/services/story_consistency_rebuild_service.py`
  - 在故事调整保存后，会执行：
    - 重新替换 `story_session_messages`
    - 重建缓存会话上下文
    - 删除当前 summary
    - 重建 session history 向量索引

这说明仓库已经认识到“故事正文修改后，记忆派生状态必须重新对齐”。

### 2.5 当前主要缺口

#### 缺口 A：没有稳定的 `memory_updates` 契约

当前后端只返回：

- `activation_logs`
- `summary_memory_snapshot`
- `contexts`

但这些字段更像调试负载，而不是稳定的“记忆更新事件”契约。

表现为：

- `retrieval.py` 中的日志结构主要围绕 `source / entry_name / score / rrf_score`
- `orchestrator.py` 中追加的是 `summary_memory applied`
- `summary_helpers.py` 中追加的是 `summary_memory updated`
- `graph/story_v2/nodes.py` 中又会补 `script_design requested`、`story_state updated`

字段形态不统一，语义也混杂“检索命中”“状态应用”“写入更新”“调试补充”。

#### 缺口 B：非流式 summary 可见性不稳定

当前非流式路径在 `StoryGenerator.generate_story(...)` 中把 summary 更新设为 `async`，而 `run_post_generation_updates(...)` 在异步任务提交后立刻返回 `get_summary_snapshot(...)`。

这会导致：

- 本轮已经触发 summary 更新
- 但响应里的 `summary_memory_snapshot` 仍可能是旧值或 `null`

即“行为已生效，响应未同步反映”。

#### 缺口 C：rollback / regenerate 没有完成 semantic/vector 对齐

- `story_rag_service/api/v2/story/session_routes.py`
  - `delete_last_message(...)` 只删除持久化消息。
  - `regenerate_last_response(...)` 只删除最后 assistant 消息并重新生成。

当前没有显式处理：

- summary stale 标记
- summary 重算
- history 向量索引重建或增量修复

这意味着在高频回滚/重生场景下，原始消息、summary 和记忆索引之间存在漂移风险。

#### 缺口 D：前端只显示快照，不显示更新过程

- `frontend/src/components/story/StoryMemorySidebar.vue`
  - 当前只展示“摘要记忆”和“命中设定”。
- `frontend/src/domains/story/composables/useStoryGeneration.ts`
  - 当前只把 `summary_memory_snapshot` 保存到 `storySessionStore`。
- `frontend/src/stores/storySession.ts`
  - 当前只保存 summary map 和 branch tree。

前端没有：

- memory update timeline
- before / after diff
- summary reset / history reindex / stale 等事件展示

#### 缺口 E：前端对命中设定的解析与后端实际字段不匹配

- `useStoryGeneration.ts` 中 `formatActivationHits(...)` 依赖 `selection_mode === explicit | rag`
- 但后端当前主要返回的是 `source = rule | vector | hybrid | bm25`

结果是：

- 后端明明有命中记录
- 前端仍可能显示为空

### 2.6 既有验证文档的补充事实

`story_rag_service/docs/TestDoc/MemoryPlan_SmokeAPI_Validation_2026-03-16.md` 已记录：

- 数据库层能看到 `story_session_messages` 与 `conversation_summaries` 的更新
- 但 API 层的 `activation_logs / contexts / summary_memory_snapshot` 并不总是稳定返回

这进一步说明当前最大问题不是“链路不存在”，而是“缺乏稳定的观测与展示契约”。

### 3.1 本期范围

- 定义 `memory_updates` 的后端事件契约。
- 设计 `rollback / regenerate / rebuild` 的一致性治理策略。

- 不重做整个故事页布局。

## 4. 核心设计原则

### 4.1 事件契约与调试日志分离

- `activation_logs` 保留为兼容调试字段。
- `memory_updates` 新增为产品级契约字段。
- 任何需要前端长期依赖的记忆状态变化，都应通过 `memory_updates` 返回，而不是要求前端推断 `activation_logs`。

### 4.2 真相源优先，派生状态可重建

- `story_session_messages` 是 episodic 真相源。
- `conversation_summaries` 是 semantic 真相源。
- 向量 history index 是派生检索索引。
- 当派生状态与真相源可能不一致时，应优先：
  - 标记 stale
  - 局部重建
  - 或显式 reset

而不是继续对外返回旧快照。

### 4.3 事件必须可追踪、可解释

对用户、前端和运维来说，“记忆发生了什么”至少要回答：

- 哪一层 memory 发生了变化？
- 是新增、合并、覆盖、失效还是重建？
- 由哪一轮输入或哪次系统动作触发？
- 变化前后是什么？
- 这次变化最终是否提交成功？

### 4.4 前端展示以“本轮变化”为先

首期前端不追求完整记忆浏览器，而是先完成：

- 最近一轮的更新事件
- 最近一轮的 summary diff
- 关键维护事件，如 `summary_reset` / `history_reindexed`

## 5. `memory_updates` 事件契约设计

## 5.1 目标

让后端每次生成、回滚、重生、重建都能输出统一的结构化事件，供：

- 故事页侧边栏展示
- 控制台分析页展示
- smoke / observability 验证
- 后续失败补偿与诊断

## 5.2 推荐字段

```ts
type MemoryUpdateEvent = {
  event_id: string
  session_id: string
  memory_layer: 'episodic' | 'semantic' | 'profile' | 'procedural' | 'system'
  action:
    | 'created'
    | 'merged'
    | 'updated'
    | 'reset'
    | 'reindexed'
    | 'superseded'
    | 'marked_stale'
    | 'rebuilt'
  source: 'generate' | 'rollback' | 'regenerate' | 'story_adjustment_commit' | 'rebuild_service'
  source_turn?: number | null
  memory_key?: string | null
  title: string
  reason?: string | null
  before?: Record<string, Any> | null
  after?: Record<string, Any> | null
  status: 'committed' | 'failed' | 'stale'
  committed_at: string
}
```

## 5.3 事件分层建议

### episodic

用于表达：

- raw messages persisted
- history index rebuilt
- history index updated

示例：

- `memory_layer=episodic, action=updated, title='原始会话消息已写入'`
- `memory_layer=episodic, action=reindexed, title='历史向量索引已重建'`

### semantic

用于表达：

- summary created
- summary merged
- summary reset
- summary marked stale

示例：

- `memory_layer=semantic, action=merged, title='摘要记忆已合并'`
- `memory_layer=semantic, action=marked_stale, title='摘要记忆已标记为待重建'`

### system

用于表达维护性动作：

- rollback 导致语义状态失效
- rebuild 完成
- commit 后触发一致性修复

## 5.4 与现有响应的兼容策略

首期建议：

- 继续保留 `activation_logs`
- 继续保留 `summary_memory_snapshot`
- 新增 `memory_updates: List[MemoryUpdateEvent]`

兼容关系：

- `activation_logs` 继续给开发调试和旧前端代码使用
- `memory_updates` 成为新前端展示和新 smoke 的唯一稳定契约

## 5.5 是否引入 journal

推荐分两级：

### Phase A：先返回内存态事件

- 在 `MemoryUpdateService`
- `StoryConsistencyRebuildService`
- `session_routes` 的 rollback/regenerate 路径

统一生成并返回 `memory_updates`。

### Phase B：再补 `memory_update_journal`

如果后续要支持：

- 前端查看跨轮次 timeline
- 后端故障补偿
- 事件失败追踪

则增加 SQLite journal 表：

```sql
memory_update_journal(
  event_id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  memory_layer TEXT NOT NULL,
  action TEXT NOT NULL,
  source TEXT NOT NULL,
  source_turn INTEGER,
  memory_key TEXT,
  title TEXT NOT NULL,
  reason TEXT,
  before_json TEXT,
  after_json TEXT,
  status TEXT NOT NULL,
  committed_at TEXT NOT NULL
)
```

本期规划建议把 journal 作为“推荐实现”，但不要阻塞第一阶段事件契约落地。

## 6. rollback / regenerate 一致性治理方案

## 6.1 当前问题

当前：

- `delete_last_message(...)` 删除的是持久化消息
- `regenerate_last_response(...)` 删除最后 assistant 后直接重新生成

缺少对：

- semantic summary
- episodic history index

的一致性治理。

## 6.2 一致性目标

在以下场景中，保证 SQLite 真相源与派生状态不长期漂移：

1. 用户删除最后一轮消息
2. 用户重新生成最后一轮
3. 故事调整保存后重建上下文

## 6.3 建议策略

### 策略 A：rollback 时显式标记 semantic stale

当删除最后一条消息或最后 assistant 消息时：

- 不强求同步重算 summary
- 但必须：
  - 产生一条 `memory_updates`
  - 标记当前 summary 为 `stale`
  - 或直接 reset 当前 summary

推荐首版做法：

- rollback 后对当前 session 产生：
  - `semantic / marked_stale`
  - `episodic / updated`

这样至少不会继续把旧 summary 当成可信快照暴露给前端。

### 策略 B：regenerate 前或 regenerate 后补一次轻量重建

推荐两种可选实现：

#### 方案 1：轻量一致性修复

- 删除 assistant message 后
- 立即清理受影响的 history index 末尾条目
- 将 summary 标记为 stale
- 新生成完成后由正常写路径重新推进

优点：

- 改动小
- 与现有 `MemoryUpdateService` 兼容

缺点：

- 某些异常中断情况下，stale summary 仍会短暂存在

#### 方案 2：复用 `StoryConsistencyRebuildService`

- rollback / regenerate 前后对当前 session 做一次小范围 rebuild

优点：

- 一致性最强

缺点：

- 开销更大
- 对常规 regenerate 可能偏重

本期建议优先采用“轻量修复 + 明确 stale 标记”，先控制复杂度。

### 策略 C：故事调整提交继续沿用 rebuild 思路，但要补事件输出

当前 `StoryConsistencyRebuildService` 已经做：

- 替换 `story_session_messages`
- reset summary
- rebuild history index

缺的不是机制，而是事件化结果。

建议补充：

- `semantic / reset`
- `episodic / rebuilt`
- `episodic / reindexed`
- `system / rebuilt`

使前端和 smoke 能看见“这次保存触发了哪些记忆修复动作”。

## 6.4 一致性验收口径

在 rollback / regenerate / story adjustment commit 之后，需要同时验证：

1. `story_session_messages` 是否与预期内容一致
2. `conversation_summaries` 是否：
   - 被标记 stale
   - 或被 reset
   - 或被重建到新版本
3. history vector index 是否：
   - 已增量更新
   - 或被重新索引
4. API 返回的 `memory_updates` 是否准确反映上述动作

## 7. 前端从“快照”升级到“事件 + diff”的方案

## 7.1 当前前端结构

当前相关模块：

- `frontend/src/components/story/StoryMemorySidebar.vue`
- `frontend/src/domains/story/composables/useStoryGeneration.ts`
- `frontend/src/stores/storySession.ts`
- `frontend/src/domains/analytics/components/AnalyticsStoryMemoryPanel.vue`
- `frontend/src/domains/story/api/storyGenerationApi.ts`

这些模块已经能展示：

- summary snapshot
- lore hit result

但不能展示：

- 某轮新增了哪些记忆
- 哪些记忆被合并
- 哪些记忆被 reset / reindexed
- summary 在本轮是如何变化的

## 7.2 首期展示目标

### 故事页侧边栏

在现有 `StoryMemorySidebar` 中新增一个区域：

- 名称建议：`记忆更新`

展示最近一轮 `memory_updates`：

- 按事件顺序展示
- 用颜色区分：
  - `created`
  - `merged`
  - `updated`
  - `reset`
  - `reindexed`
  - `marked_stale`

每项至少显示：

- 标题
- layer
- action
- reason
- 时间

### Summary Diff

在“摘要记忆”区域补一个 diff 视图：

- 新增 facts
- 删除或失效 facts
- summary_text 是否变化

首版可以不做复杂文本 diff，只做：

- `new_facts`
- `removed_facts`
- `summary_changed: true/false`

### 控制台分析页

在 `AnalyticsStoryMemoryPanel` 中增加：

- 每个会话最近一次更新类型
- 更新次数统计
- `summary reset / reindexed / stale` 这类维护事件计数

## 7.3 推荐前端数据结构

在 `storyGenerationApi.ts` 中新增：

```ts
type MemoryUpdateEvent = {
  event_id: string
  session_id: string
  memory_layer: 'episodic' | 'semantic' | 'profile' | 'procedural' | 'system'
  action: string
  source: string
  source_turn?: number | null
  memory_key?: string | null
  title: string
  reason?: string | null
  before?: Record<string, unknown> | null
  after?: Record<string, unknown> | null
  status: 'committed' | 'failed' | 'stale'
  committed_at: string
}
```

在 `storySessionStore` 中新增：

- `memoryEventMap: Record<sessionId, MemoryUpdateEvent[]>`
- `lastSummaryDiffMap: Record<sessionId, SummaryDiff>`

## 7.4 前端交互建议

### 侧边栏默认折叠粒度

默认只展示：

- 最近 5 条记忆事件
- 最近一轮 summary diff

避免侧边栏过长。

### 事件详情展开

点击单条事件时展示：

- `before`
- `after`
- `source_turn`

这样前端既能简洁，也保留足够诊断信息。

## 7.5 兼容期策略

前端迁移期应按以下顺序：

1. 先修正 `activation_logs` 命中解析逻辑，别再依赖 `selection_mode`
2. 再接入 `memory_updates`
3. 最后增加 diff 可视化

原因：

- 当前命中设定区域已经存在明显字段错配
- 若不先修复，用户会误以为“事件化改造后也还是没有命中”

## 8. 分阶段实施计划

## Phase 1：定义 `memory_updates` 契约

### 目标

- 后端每次生成、回滚、重生、重建都能产出统一事件。
- API 响应新增 `memory_updates` 字段。
- 保持 `activation_logs` 兼容。

### 主要改动面

- `story_rag_service/api/v2/schemas.py`
- `story_rag_service/application/memory/update_service.py`
- `story_rag_service/services/story_generation/summary_helpers.py`
- `story_rag_service/services/story_consistency_rebuild_service.py`
- `story_rag_service/graph/story_v2/nodes.py`

### 验收

- 正常生成后，能返回至少一条 `episodic` 或 `semantic` 事件。
- 故事调整保存后，能返回 `reset / rebuilt / reindexed` 事件。
- 旧字段仍可用。

## Phase 2：补齐 rollback / regenerate 一致性治理

### 目标

- 删除消息与重生成不再只改 raw messages。
- summary 与 history index 至少进入可解释、可恢复的状态。

### 主要改动面

- `story_rag_service/api/v2/story/session_routes.py`
- `story_rag_service/services/story_consistency_rebuild_service.py`
- 可选：增加轻量 session rebuild helper

### 验收

- rollback 后不会继续向前端暴露“已过时但未标识”的 summary。
- regenerate 后 summary / vector 状态最终回到一致。
- API 返回的 `memory_updates` 能解释整个过程。

## Phase 3：前端事件 + diff 可视化

### 目标

- 故事页能看见最近一轮记忆发生了什么。
- 控制台能看见跨会话的记忆演化情况。

### 主要改动面

- `frontend/src/domains/story/api/storyGenerationApi.ts`
- `frontend/src/domains/story/composables/useStoryGeneration.ts`
- `frontend/src/stores/storySession.ts`
- `frontend/src/components/story/StoryMemorySidebar.vue`
- `frontend/src/domains/analytics/components/AnalyticsStoryMemoryPanel.vue`

### 验收

- 本轮生成后，故事页侧边栏能展示最近一轮 `memory_updates`
- summary diff 能展示 facts 增减
- 命中设定区域不再依赖错误的 `selection_mode`

## 9. 风险与控制策略

### 风险 1：过度设计为新的 memory 框架

影响：

- 与现有 `MemoryOrchestrator / MemoryUpdateService` 重叠
- 增加迁移成本

控制：

- 本期只加事件契约和一致性治理，不重做四层模型

### 风险 2：事件字段过重

影响：

- 响应变大
- journal 膨胀

控制：

- `before / after` 只存结构化摘要，不存大段原文
- 首版优先展示 facts 与状态摘要，不做全文 patch

### 风险 3：rollback 路径为了强一致做了过重 rebuild

影响：

- 请求变慢

控制：

- 首版优先采用“标记 stale + 增量修复”
- 全量 rebuild 只留给 story adjustment commit 这类重操作

### 风险 4：前端一次性改太多造成现有 sidebar 回归

影响：

- 故事页可读性下降

控制：

- 保持现有摘要区域
- 新增“记忆更新”分区而不是重写整个侧栏

---

## 10. 验证与验收方案

## 10.1 结构验证

检查是否满足：

- `memory_updates` 是独立稳定契约
- `activation_logs` 仍保留兼容
- rollback / regenerate 有显式 semantic / vector 一致性策略
- 前端不再依赖 `selection_mode`

## 10.2 行为验证

重点验证场景：

1. 普通生成
   - `story_session_messages` 增加
   - `memory_updates` 返回 episodic / semantic 事件

2. 非流式生成
   - 即使 summary 仍异步更新，也必须有事件明确说明：
     - 已触发更新
     - 或当前快照仍为旧值

3. rollback
   - 删除消息后，summary 不再以“看似有效”的旧状态暴露
   - `memory_updates` 至少出现 `marked_stale` 或 `reset`

4. regenerate
   - 新一轮生成后，summary / history index 状态重新回正

5. story adjustment commit
   - 返回 `reset / rebuilt / reindexed` 事件

## 10.3 前端验证

1. 故事页最近一轮生成后：
   - 能看到记忆更新时间线
   - 能看到 summary diff

2. 命中设定区域：
   - 能正确显示 `rule / vector / hybrid / bm25` 命中

3. 控制台：
   - 能看到会话级 memory 更新计数与最近更新时间

---

## 11. 建议执行顺序

推荐严格按以下顺序推进：

1. 先定义 `memory_updates` 事件契约与响应 schema。
2. 再补齐 rollback / regenerate 的 summary/vector 一致性治理。
3. 最后接前端的事件时间线与 summary diff 展示。

原因：

- 没有稳定事件契约，前端无法可靠展示。
- rollback / regenerate 不先治理，一旦可视化上线，会把旧 summary 漂移直接暴露给用户。
- 前端展示应建立在后端可靠事件之上，而不是继续解析调试日志。

---

## 12. 最终建议

本期最重要的设计判断是：

- 不要把问题理解为“再做一套 memory 系统”。
- 应把问题理解为“现有 memory 系统已经能跑，但缺少稳定事件契约、回退一致性治理和前端可见性”。

因此最合理的路线是：

**先把 memory update 变成一等事件，再把回退修复做实，最后把这些事件展示给前端。**

这条路线与现有架构最一致，风险最可控，也最能快速把“记忆链路存在”升级为“记忆链路可证明、可观察、可理解”。
