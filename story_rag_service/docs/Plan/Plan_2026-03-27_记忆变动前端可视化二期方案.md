# Plan_2026-03-27_记忆变动前端可视化二期方案

## 1. 背景

截至 2026-03-27，仓库已经完成“记忆事件契约 + 一致性治理 + 一期前端可视化”的第一阶段能力：

- 后端已提供稳定的 `memory_updates` 契约。
- `rollback / regenerate / story adjustment commit` 已具备显式的 episodic / semantic 一致性修复。
- `memory_update_journal` 已持久化到 SQLite。
- 前端已存在独立管理路由 `/dashboard/memory-updates`。
- 故事页侧栏已能展示最近一轮事件与 summary diff。

同时，以下验证已经为二期方案提供了事实基础：

- `story_rag_service/docs/TestResult/MemoryEventContract_Consistency_SmokeAPI_Validation_2026-03-27.md`
- `story_rag_service/docs/TestResult/SemanticScenario_SmokeAPI_Validation_2026-03-27.md`

其中，semantic 场景 smoke 已验证完整语义链路：

1. `semantic:created:generate`
2. `semantic:reset:rollback`
3. `semantic:created:generate`（regenerate 后）
4. `semantic:reset:story_adjustment_commit`

并确认 `memory_updates`、`conversation_summaries`、`memory_update_journal` 三者在语义层是一致且可审计的。

这意味着二期不需要再证明“链路是否存在”，而是要解决：

- 如何让前端把这条链路展示得更清晰、更稳定、更可解释。
- 如何让管理区从“本地即时态摘要页”升级为“服务端可审计时间线”。
- 如何让故事页侧栏从“事件列表”升级为“本轮记忆解释器”。

## 2. 当前现状与问题定位

以下结论来自当前代码，而不是推测。

### 2.1 已有前端入口

- `frontend/src/router/index.ts`
  - 已存在 `/dashboard/memory-updates` 独立路由。
- `frontend/src/components/layout/AppSidebar.vue`
  - “记忆变动”已作为管理区一级入口，与 `Lorebook`、`剧本设计` 同级。

### 2.2 已有前端页面

- `frontend/src/views/DashboardStoryMemoryView.vue`
  - 已能展示：
    - 摘要会话数
    - 最近事件数
    - 会话卡片
    - 最近摘要状态
    - 最近事件摘要
- `frontend/src/components/story/StoryMemorySidebar.vue`
  - 已能展示：
    - 最近一轮 `memory_updates`
    - `summary diff`
    - 命中设定

### 2.3 当前前端数据源仍偏“本地即时态”

- `frontend/src/stores/storySession.ts`
  - 当前 `memoryEventMap` 保存的是裁剪后的 `StoryMemoryEventRecord[]`
  - 已保留字段：
    - `eventId`
    - `sessionId`
    - `storyTitle`
    - `worldId`
    - `memoryLayer`
    - `action`
    - `source`
    - `sourceTurn`
    - `title`
    - `reason`
    - `status`
    - `committedAt`
  - 未保留字段：
    - `before`
    - `after`
    - `memory_key`

这会直接限制：

- before / after diff 可视化
- 精确解释 semantic reset / recreated
- 事件详情 drill-down

### 2.4 管理区尚未接入服务端 journal 查询

当前代码中：

- 后端已写入 `memory_update_journal`
- 但 API 层尚未看到面向前端的 journal 查询接口
- 管理区页面仍主要依赖客户端 store

这意味着当前“记忆变动”管理页本质上仍是：

- 本地缓存总览页

而不是：

- 服务端可审计时间线

### 2.5 一期可视化已经够用，但二期缺口明确

当前体验已经能满足“看见最近发生了什么”，但还不能满足：

1. 刷新页面后稳定回看完整历史
2. 跨设备一致查看记忆变动
3. 按 source / layer / status 过滤
4. 查看某个 session 的完整时间线
5. 明确区分：
   - “尚未生成摘要”
   - “摘要已 reset”
   - “摘要已重新创建”
   - “摘要当前 stale”

## 3. 二期目标

本期目标定义为：

**把现有记忆变动前端从“本地即时摘要展示”升级为“服务端可审计 + 故事页可解释”的双层可视化系统。**

### 3.1 目标一：管理区升级为可审计时间线

管理区 `/dashboard/memory-updates` 应支持：

- 全局筛选
- 按 session 查看完整时间线
- 查看 semantic 生命周期
- 直接基于服务端 journal 回放，不依赖单纯本地 store

### 3.2 目标二：故事页升级为本轮记忆解释器

故事页侧栏应支持：

- 一眼看懂本轮是否产生 summary
- 一眼看懂本轮是否 reset summary
- 一眼看懂 history index 是否重建
- 对失败态、stale 态提供强提示

### 3.3 目标三：统一 semantic 生命周期表达

前端所有页面都要用同一套语义状态：

- `absent`
- `created`
- `reset`
- `recreated`
- 预留 `stale`

避免不同页面对“为什么没有摘要”给出互相矛盾的解释。

## 4. 设计原则

### 4.1 管理区与故事页职责分离

管理区负责：

- 跨轮次
- 跨会话
- 跨设备
- 可审计时间线

故事页负责：

- 本轮即时反馈
- 本轮语义解释
- 不承担历史审计职责

### 4.2 服务端 journal 是管理区的权威数据源

管理区应优先使用：

- `memory_update_journal`

而不是：

- 客户端临时缓存

原因：

- journal 已被 smoke 验证可与 API 对账
- journal 能跨刷新、跨设备稳定存在
- journal 更适合分页、筛选、审计

### 4.3 storySessionStore 继续承担“即时态”

`storySessionStore` 不应被废弃，而应重新定位：

- 为故事页本轮展示服务
- 保留最近事件与摘要状态
- 不承担完整审计历史责任

### 4.4 不重做现有导航与页面骨架

二期应复用：

- `DashboardStoryMemoryView.vue`
- `StoryMemorySidebar.vue`
- `storySessionStore`
- `/dashboard/memory-updates` 路由

避免把已有入口和信息架构再次推翻。

## 5. 二期页面结构方案

## 5.1 管理区总览页

路由保持不变：

- `/dashboard/memory-updates`

页面建议升级为三层结构：

### A. 顶部过滤栏

字段建议：

- 关键词搜索
  - session id
  - story title
  - world id
- `source`
  - `generate`
  - `rollback`
  - `regenerate`
  - `story_adjustment_commit`
- `memory_layer`
  - `episodic`
  - `semantic`
- `status`
  - `committed`
  - `failed`
  - `stale`
- 时间范围
  - 最近 1 小时
  - 最近 24 小时
  - 最近 7 天
  - 自定义区间

### B. 中间结果区

结果区建议采用 master-detail 双栏布局：

- 左侧：session 列表 / 查询结果列表
- 右侧：当前 session 的时间线详情

左侧每条结果至少显示：

- storyTitle
- sessionId
- worldId
- 最近一次事件时间
- 最近一次 semantic 状态
- 事件总数
- 是否存在失败事件

### C. 右侧时间线详情区

展示当前 session 的完整时间线：

- 按时间倒序或按操作分组
- 每条事件显示：
  - title
  - layer
  - action
  - source
  - status
  - committed_at
- 支持展开详情：
  - before
  - after
  - memory_key
  - source_turn

详情区还应包含“摘要状态卡”：

- 当前 summary 是否存在
- 最近一次 `semantic:created`
- 最近一次 `semantic:reset`
- facts 数量
- 当前语义状态

---

## 5.2 故事页侧栏

故事页继续保留 `StoryMemorySidebar`，但职责升级为：

- 本轮记忆解释器

推荐结构：

### A. 本轮结论条

直接汇总本轮结果，例如：

- “摘要已创建”
- “摘要已重置，等待后续重建”
- “仅更新历史索引，未发生语义变化”
- “记忆修复失败，请检查重建事件”

### B. 本轮事件链

对当前这一轮的事件按顺序展示：

- `episodic:rebuilt`
- `episodic:reindexed`
- `episodic:updated`
- `semantic:created/reset`

每条支持展开详情。

### C. 摘要状态区

当前若有 summary：

- 展示摘要文本
- 展示 key facts
- 展示 summary diff

当前若没有 summary：

- 不再只显示“暂无摘要”
- 需要区分：
  - 尚未生成
  - 刚被 reset
  - stale

---

## 5.3 语义生命周期展示

建议在管理页与故事页统一使用如下语义状态：

### `absent`

含义：

- 当前没有 summary
- 且从未生成过 summary

典型场景：

- 新会话
- 对话轮次不足

建议展示：

- 中性灰色
- 文案：`尚未生成摘要`

### `created`

含义：

- 本轮刚创建 summary

典型场景：

- 第 4 轮首次触发 summary 生成

建议展示：

- 绿色或紫色强调
- 文案：`摘要已创建`

### `reset`

含义：

- summary 被明确移除

典型场景：

- rollback
- story adjustment commit

建议展示：

- 橙色或红色强调
- 文案：`摘要已重置`

### `recreated`

含义：

- summary 在 reset 之后重新创建

典型场景：

- regenerate 后再次出现 `semantic:created`

建议展示：

- 绿色强调
- 文案：`摘要已重建`

### `stale`

含义：

- summary 尚存在，但已经不可信

当前状态：

- 预留

建议展示：

- 黄色强调
- 文案：`摘要已过期，等待刷新`

## 6. API 方案

## 6.1 新增 journal 查询接口

二期建议新增至少两类查询接口。

### A. 全局列表查询

建议：

`GET /api/v2/memory-updates`

查询参数建议：

- `session_id`
- `world_id`
- `story_id`
- `source`
- `memory_layer`
- `status`
- `date_from`
- `date_to`
- `page`
- `page_size`

响应建议：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 120
}
```

### B. 单 session 时间线查询

建议：

`GET /api/v2/story/session/{session_id}/memory-updates`

响应建议：

- 返回该 session 的时间线
- 可选返回 summary 状态聚合信息

### C. 可选：摘要状态汇总接口

若前端需要更轻量的总览，可以考虑增加聚合接口：

`GET /api/v2/memory-updates/summary`

用于返回：

- 总事件数
- 有摘要的会话数
- 最近 reset 数
- 最近 failed 数

---

## 6.2 查询字段建议

二期建议继续沿用现有 `MemoryUpdateEvent`，并评估补充以下字段：

- `operation_id`
  - 用于把同一次 rollback / regenerate / commit 形成的事件链归组
- `sequence`
  - 用于表示操作内顺序
- `display_kind`
  - 例如：
    - `write`
    - `repair`
    - `semantic_update`

说明：

- 这不是二期必须项
- 但如果缺失，前端只能按 `committed_at` 近似分组
- 会降低时间线解释能力

## 7. Store 方案

## 7.1 现状问题

当前 `storySessionStore` 会把事件裁剪成 `StoryMemoryEventRecord`，导致：

- 无法展示 `before / after`
- 无法展示 `memory_key`
- 无法做更强的详情 drill-down

## 7.2 二期建议：拆成两层

### A. runtime event cache

职责：

- 服务故事页与故事调整页
- 只关心最近一轮或最近若干轮
- 提供本轮即时可视化

特点：

- 可保留在 Pinia
- 可保留有限持久化
- 应保留完整 `MemoryUpdateEvent`

### B. query result cache

职责：

- 服务管理区
- 承接服务端 journal 查询结果
- 支持分页、筛选、详情查看

特点：

- 不建议长期持久化全量数据
- 更适合按需缓存
- 应与 runtime cache 明确分离

---

## 7.3 store 字段建议

建议新增或调整：

- `runtimeMemoryEventsBySession`
- `runtimeSummaryStateBySession`
- `memoryTimelineQuery`
  - 当前过滤条件
  - 当前分页信息
  - 当前结果列表
- `memoryTimelineDetailBySession`
  - 当前已加载的 session 时间线

同时，保留：

- `summaryMap`
- `branchTrees`

## 8. 交互状态设计

## 8.1 管理区

必须覆盖：

- `loading`
- `empty`
- `error`
- `partial_data`
- `filtered`
- `selected_session`
- `no_summary`
- `summary_reset_recently`
- `failed_event_present`

### 空态

当 journal 无记录时：

- 文案不应写成“暂无摘要”
- 应写成：
  - `尚未产生任何记忆变动事件`

### 错误态

当查询失败时：

- 展示重试按钮
- 保留当前筛选条件

### 详情空态

当 session 有事件但当前无 summary 时：

- 区分：
  - “该 session 尚未生成过摘要”
  - “该 session 的摘要刚被 reset”

---

## 8.2 故事页

故事页侧栏必须覆盖：

- `this_round_created`
- `this_round_reset`
- `this_round_recreated`
- `this_round_only_episodic`
- `this_round_failed`

推荐交互：

- 默认折叠显示本轮结论
- 展开可查看事件详情
- reset 事件发生时，摘要区显示原因说明而不是纯空白

## 11. 风险与控制

### 风险 1：管理区继续依赖本地 store

影响：

- 刷新不稳定
- 跨设备不可审计
- 长时间线不可分页

控制：

- 管理区优先改为读取服务端 journal 查询接口

### 风险 2：事件仍被裁剪

影响：

- 无法展示 before/after
- 无法解释 semantic reset/recreated

控制：

- 前端保留完整 `MemoryUpdateEvent`

### 风险 3：没有操作分组字段

影响：

- 前端只能按时间近似推断同一次操作链路

控制：

- 在二期 API 设计阶段优先评估 `operation_id`

### 风险 4：不同页面文案不一致

影响：

- 用户不知道“暂无摘要”究竟是没生成还是刚被 reset

控制：

- 先定义统一 semantic 生命周期规则，再做页面实现

## 12. 验收标准

二期完成后，应满足以下标准：

1. 管理区能从服务端 journal 查询记忆事件，而不是只读本地 store。
2. 管理区能按 `source / layer / status / time range` 查询与过滤。
3. 管理区能查看 session 级完整时间线。
4. 故事页能明确解释本轮是 `semantic:created`、`semantic:reset`、`recreated` 还是仅 episodic 更新。
5. 前端可区分：
   - 尚未生成摘要
   - 摘要已重置
   - 摘要已重新创建
   - 摘要已 stale（预留）
6. semantic smoke 文档中的链路在前端页面上能被逐步验证。

---

## 13. 最终建议

这次二期最关键的判断是：

- 不要继续把“记忆变动”当成故事页的一个附属小组件。
- 应把它升级为：
  - 管理区中的可审计事件系统
  - 故事页中的本轮解释系统

最合理的执行顺序是：

1. 先补服务端查询 API。
2. 再升级 store。
3. 然后做管理区时间线。
4. 再做故事页本轮解释器。
5. 最后用 semantic 生命周期统一规则和验收清单收口。

这样可以在不推翻现有结构的前提下，把当前已验证通过的后端语义能力真正转化为可理解、可追踪、可前端消费的产品能力。

## 14. 推荐实施顺序（细化版）

为避免同时改动 API、store、管理页和故事页带来的回归风险，推荐按以下切片推进：

### Slice 1：服务端 journal 查询与管理区接入

目标：

- 管理区摆脱“只读本地 store”的限制
- 先具备服务端可审计的事件列表与 session 时间线

范围：

- 新增 journal 查询 API
- 前端新增 memory updates query hooks
- 管理区接入服务端数据源

本切片不做：

- 故事页侧栏二期增强
- stale 的复杂交互推导
- operation_id 分组增强

### Slice 2：store 模型升级

目标：

- 保留完整事件字段
- 明确 runtime cache 与 query cache 的边界

范围：

- `storySessionStore` 重构
- 现有故事页与故事调整页写入链路适配

### Slice 3：故事页本轮解释器

目标：

- 让用户在故事页中直接理解“本轮记忆发生了什么”

范围：

- `StoryMemorySidebar` 升级
- 本轮 summary created/reset/recreated 的解释视图

### Slice 4：semantic 生命周期统一与验收闭环

目标：

- 管理区与故事页文案/颜色/状态一致
- semantic smoke 结果能映射到前端验收
