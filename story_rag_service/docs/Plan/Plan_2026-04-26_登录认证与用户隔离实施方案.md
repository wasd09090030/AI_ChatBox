# 登录认证与用户隔离实施方案

## 日期

- 本地日期：2026-04-26

## 背景

- 当前前端通过 `frontend/src/domains/user/api/userIdentity.ts` 在 `localStorage` 生成 `storybox_user_id`，并向后端透传 `X-User-ID`。
- 当前后端 `users` / `user_settings` 主要用于模型配置与 API Key 存储，不构成真实认证。
- 当前 `worlds`、`stories`、`story_sessions`、`script_designs`、`story_runtime_states` 等核心业务资源未稳定绑定 `owner_user_id`，不能保证用户业务数据隔离。

## 目标

- 为项目补齐最小可落地的账号密码登录能力。
- 不引入角色权限区分，所有用户能力一致。
- 业务数据必须按用户隔离，禁止通过伪造请求头读取他人资源。
- 兼容已有匿名 `storybox_user_id` 存量数据，提供一次性认领迁移。

## 关键决策

### 认证方式

- 浏览器端使用 `HttpOnly Cookie` 保存会话，不再把真实身份长期暴露给前端 JavaScript。
- 后端通过 `current_user` 依赖解析登录态，业务接口统一读取当前会话用户。

### 用户账户模型

- 在现有 `users` 聚合上扩展：
  - `login_identifier`
  - `display_name`
  - `password_hash`
  - `status`
  - `last_login_at`
- 继续复用现有 `user_settings` 作为用户配置子对象。

### 会话模型

- 新增 `auth_sessions` 表。
- Cookie 内仅保存随机 session token。
- 数据库存储 token 摘要、过期时间、用户归属与审计元数据。

### 数据隔离模型

- 一级业务资源直接增加 `owner_user_id`：
  - `worlds`
  - `stories`
  - `script_designs`
  - `story_sessions`
  - `story_runtime_states`
- 高频二级资源补齐归属链或冗余 owner：
  - `story_session_messages`
  - `lorebook_entries`
  - `memory_update_journal`
  - `entity_state_events`

### 兼容策略

- 保留 `storybox_user_id` 仅作为一次性 legacy 认领桥，不再作为真实身份源。
- 登录成功后检测本地 legacy id，调用后端 claim 接口迁移可归属资源。
- 认领流程必须幂等、可审计、不可长期复用为后门。

## 前端设计方向

- 登录页采用“编辑台 / 手稿页”风格，不复用主工作台侧栏布局。
- 桌面端双栏：
  - 左侧品牌叙事区：大标题、简短引导、纹理背景。
  - 右侧认证表单区：登录 / 注册切换、错误提示、认领提示。
- 移动端收敛为单栏，保留视觉识别但优先保证可用性。

## 阶段划分

### 阶段 1：后端认证基础设施

- 扩展用户表结构与模型。
- 新增认证服务、认证仓储、认证依赖。
- 新增 `/api/v2/auth/register`、`/login`、`/logout`、`/me`。
- 修正 CORS 与 Cookie 配置。

### 阶段 2：核心业务资源 owner 迁移

- 为核心表补齐 `owner_user_id` 与索引。
- 改造仓储与服务签名，使 owner 成为显式约束。
- 同步更新运行时自动建表与 Alembic 迁移。

### 阶段 3：业务路由鉴权收口

- `world/story/session/memory/provider/script/lorebook` 路由统一切到 `current_user`。
- 不再信任外部自由传入的 `X-User-ID`。
- 故事生成链路保留内部 `user_id` 参数，仅作为后端上下文传递。

### 阶段 4：前端登录页与会话恢复

- 新增登录页、认证 API、认证状态仓库。
- `App.vue` 启动时恢复会话。
- `router.beforeEach` 增加登录守卫与 redirect 回跳。
- Axios 改为凭证模式。

### 阶段 5：legacy 数据认领

- 登录后读取本地 legacy id。
- 调用 claim 接口执行资源 owner 迁移。
- 迁移完成后清理 legacy 标记。

### 阶段 6：验证与文档

- 补充 smoke 验证：
  - register / login / me / logout
  - owner 隔离
  - legacy claim
- 手工验证前端跳转、刷新恢复、登出与多用户隔离。
- 更新 `PROJECT_STRUCTURE.md` 与仓库根 `doc/` 任务记录。

## 风险

- 最大风险是旧匿名数据认领错误，必须保证幂等与可追溯。
- 若只做登录页而不做 owner 迁移，会形成“已登录但仍可串数据”的伪安全状态。
- 当前故事链路和记忆链路较长，session / story / runtime / memory 的 owner 约束必须联动检查。
