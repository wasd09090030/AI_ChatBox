# plan_2026-03-18_控制台目录与场景化模型选择重构方案

## 1. 背景与目标

本方案针对当前前端“管理”分组中的两个页面进行信息架构重构与能力拆分：

- `frontend/src/views/DashboardAnalyticsView.vue`
- `frontend/src/views/DashboardConfigView.vue`

目标不是简单改名，而是把这两个页面从现有“管理”语义中独立出来，放入新的“控制台”目录下，并完成页面职责重组。

### 1.1 目标拆解

本次规划要同时达成以下 4 项目标：

1. 侧边栏新增“控制台”分组，原本属于“管理”语义的 Analytics / Config 相关能力迁移至控制台。
2. `DashboardAnalyticsView.vue` 拆为两个页面：
   - Token 相关分析页
   - 其它综合分析页
3. `DashboardConfigView.vue` 拆为两个页面：
   - 模型管理页：保留现有 AI 提供商配置 + 生成参数区块，尽量不改交互与布局
   - 大模型选择页：允许分别为三个场景自由选择模型，并由后端记录
4. 三个独立场景的模型偏好需要落到后端持久化：
   - 故事生成
   - Prompt 优化（对应当前“AI 增强 Prompt”能力）
   - 故事调整

### 1.2 本次规划的核心原则

- 优先复用现有页面内的查询逻辑、图表组件、Provider 配置组件与生成参数组件。
- 不在本期顺手重构无关模块，避免把“页面拆分”升级成大面积设置中心重构。
- 场景化模型偏好必须以后端为真相源，而不是继续只停留在前端本地存储。
- 保持旧行为可兼容：未配置场景模型时，仍能按当前默认模型逻辑工作。

## 2. 现状基线与代码事实

以下内容均来自当前仓库代码，不依赖假设。

### 2.1 路由与导航现状

#### 路由现状

`frontend/src/router/index.ts` 当前包含以下相关路由：

- `/dashboard/config` -> `DashboardConfigView.vue`
- `/dashboard/analytics` -> `DashboardAnalyticsView.vue`
- `/dashboard/lorebook` -> `DashboardLorebookView.vue`
- `/dashboard/scripts` -> `DashboardScriptDesignView.vue`

同时，当前路由 meta 中：

- `DashboardConfigView.vue` 的 `group` 为 `dashboard`
- `DashboardAnalyticsView.vue` 的 `group` 为 `dashboard`

#### 侧边栏现状

`frontend/src/components/layout/AppSidebar.vue` 当前使用硬编码分组：

- “功能”组：`/story`、`/story-adjustment`
- “管理”组：
  - `/dashboard/config`
  - `/dashboard/lorebook`
  - `/dashboard/scripts`
  - `/dashboard/analytics`

结论：

1. “管理”不是从路由自动生成，而是 sidebar 内手工维护。
2. 要把 Config / Analytics 从“管理”中独立出来，必须同时改路由命名与 sidebar 分组。
3. Lorebook / 剧本设计页是否跟随迁移，需要单独决策；本次需求只明确要求 Config / Analytics 独立出来，因此不建议扩大范围。

- `selectedModel`

#### 页面内指标卡

当前 metrics 同时混合了两类内容：

1. token 类指标
   - `Input Token`
   - `Output Token`
   - `单次总消耗`
2. 非 token 类指标
   - `总请求`
   - `成功率`
   - `失败次数`
   - `平均耗时`
   - `设定命中`

#### 页面内图表 / 面板

- `RequestsTrendChart`
- `TokenBreakdownTrendChart`
- `TokenSourceShareChart`
- `LatencyTrendChart`
- `ModelDistributionChart`
- `WorldDistributionChart`
- `EventTypeDistributionChart`
- `AnalyticsEventsTable`
- `AnalyticsStoryMemoryPanel`
- `AnalyticsInsightPanel`
- `AnalyticsMetricGrid`

结论：

1. Analytics 目前已经具备足够细粒度的组件，适合“重组页面”，不适合复制业务逻辑。
2. AnalyticsHeader 和 query 组合逻辑是两个新页面最明显的复用点。
3. 页面拆分的重点不在 API，而在页面编排和共享过滤状态的抽取。

### 2.3 Config 页面现状

`frontend/src/views/DashboardConfigView.vue` 当前实际由三个区块构成：

1. `ProviderConfigCards`
   - AI 提供商
   - API Key 管理
   - Base URL 管理
   - 模型列表获取
   - 当前 provider 的模型选择
   - 设为默认 provider/model
2. `GenerationParamsPanel`
   - `temperature`
   - `maxTokens`
3. `ConfigResetPanel`
   - 恢复默认配置

该页面的数据与交互主要来自 `frontend/src/domains/settings/composables/useDashboardConfig.ts`。

### 2.4 前端配置存储现状

`frontend/src/stores/config.ts` 当前维护的是前端侧长期配置，核心字段为：

- `defaultProvider`
- `defaultModel`
- `temperature`
- `maxTokens`
- `providerBaseUrls`
- `apiKeys`（仅前端保存 masked preview）

持久化方式：

- `defaultProvider/defaultModel/temperature/maxTokens` 通过 `frontend/src/domains/settings/api/configStorageApi.ts` 与 `storage` 层同步
- provider API key 实际通过后端 `/api/v2/providers/config` 保存
- provider base_url 通过后端 `/api/v2/providers/config` 保存，同时前端也保留本地状态

结论：

1. 当前“默认模型”主要是前端本地配置概念。
2. 当前“提供商配置”是真正落后端的。
3. 如果新增“故事生成 / Prompt 优化 / 故事调整”的独立模型选择，不能只继续塞进 `configStore.config.defaultModel`。

### 2.5 后端用户设置现状

`story_rag_service/models/user.py` 中，`UserSettings` 当前只包含：

- `default_model`
- `temperature`
- `max_tokens`
- 各 provider 的 API key
- 各 provider 的 base_url

`story_rag_service/services/database.py` 中的 `user_settings` 表当前也只包含：

- `default_model`
- `temperature`
- `max_tokens`
- provider key / base_url 相关列

没有任何场景化模型偏好字段。

### 2.6 后端 provider / 模型接口现状

当前后端与模型配置相关的 API 主要在：

- `story_rag_service/api/v2/provider_routes.py`

已存在接口：

- `GET /api/v2/providers`
- `POST /api/v2/providers/config`
- `POST /api/v2/providers/test-connection`
- `GET /api/v2/providers/{provider}/models`

这些接口主要覆盖：

- provider 可用性
- API key 保存
- Base URL 保存
- 连通性测试
- 获取 provider 的模型列表

但并不保存“某个业务场景应该用哪个 provider/model”。

### 2.7 三个模型消费入口的现状

#### 故事生成

前端：

- `frontend/src/domains/story/composables/useStoryGeneration.ts`

当前调用故事生成时，会显式带上：

- `provider: configStore.config.defaultProvider`
- `model: configStore.config.defaultModel`
- `base_url: configStore.providerBaseUrls[defaultProvider]`
- `temperature: configStore.config.temperature`

后端：

- `story_rag_service/api/v2/story/generation_routes.py`
- `story_rag_service/services/story_generation/llm_factory.py`

若请求未显式提供 model，则后端会尝试从 `user.settings.default_model` 回退。

#### Prompt 优化

当前仓库中所谓“Prompt 优化”并不是一个单独页面，而是 `StoryView.vue` 内的“AI 增强 Prompt”能力。

前端入口：

- `frontend/src/domains/story/composables/useStoryGeneration.ts`
- `previewEnhancedStoryInput(...)`

后端入口：

- `POST /api/v2/story/input-enhancement/preview`
- `StoryGenerator.preview_enhanced_input(...)`

当前该入口也显式传：

- `provider`
- `model`
- `base_url`
- `temperature`

且默认仍来自 `configStore.config.defaultProvider/defaultModel`。

#### 故事调整

前端入口：

- `frontend/src/views/StoryAdjustmentView.vue`

后端入口：

- `story_rag_service/services/story_adjustment_service.py`

当前故事调整页调用润色时，也显式使用：

- `configStore.config.defaultProvider`
- `configStore.config.defaultModel`
- 对应 provider 的 `base_url`

结论：

1. 三个场景目前全部共享前端的单一默认 provider/model。
2. 即便后端增加场景化偏好，如果前端仍然继续每次显式传默认 provider/model，后端的新配置也不会生效。
3. 因此“后端记录”不是单独后端任务，还必须伴随前端请求构造逻辑调整。

## 3. 需求解释与范围界定

### 3.1 本期明确在范围内

1. 新增“控制台”导航分组。
2. 将原 Config / Analytics 能力从“管理”中独立出来。
3. 拆分 Analytics 为两个页面。
4. 拆分 Config 为两个页面。
5. 为三个场景新增后端持久化模型偏好。
6. 让三个模型消费入口接入新的场景偏好。
7. 保持旧用户、旧链接、旧默认模型逻辑的兼容。

### 3.2 本期明确不在范围内

1. 不重做 Lorebook 或剧本设计页本身的交互逻辑。
2. 不把“Prompt 优化”扩展成一个新的独立页面。
3. 不重写 ProviderConfigCards 的整体设计语言。
4. 不在本期引入复杂的通用设置中心、权限系统或多租户配置层。
5. 不重构 analytics 后端接口或统计口径。

### 3.3 关键解释：Prompt 优化的真实对应关系

为了避免开发时理解偏差，本方案明确：

- 产品文案中的“Prompt 优化”
- 实现上对应当前 `StoryView.vue` 中的“AI 增强 Prompt”
- 后端接口对应 `POST /api/v2/story/input-enhancement/preview`

也就是说，本期“大模型选择页”中的“Prompt 优化模型”，不是一个未来概念，而是当前已存在能力的场景化模型选择。

## 4. 信息架构与路由重构方案

## 4.1 分组策略

推荐采用三组并存，而不是强行把所有后台页都迁移到控制台：

- 功能
  - 故事创作
  - 故事调整
- 管理
  - Lorebook
  - 剧本设计
- 控制台
  - Token 分析
  - 综合分析
  - 模型管理
  - 模型选择

这样做的原因：

1. 与需求精准匹配：只把 Config / Analytics 从“管理”中独立出来。
2. 变更更小：Lorebook / 剧本设计现有路径和认知不受冲击。
3. 语义更清晰：
   - “管理”偏内容资产管理
   - “控制台”偏观测、配置、运行控制

## 4.2 推荐路由命名

推荐新增以下路由：

- `/console/analytics/tokens`
- `/console/analytics/overview`
- `/console/models/management`
- `/console/models/selection`

推荐路由名称：

- `console-analytics-tokens`
- `console-analytics-overview`
- `console-model-management`
- `console-model-selection`

### 兼容路由建议

保留旧地址作为 redirect：

- `/dashboard/analytics` -> `/console/analytics/overview`
- `/dashboard/config` -> `/console/models/management`

这样可以：

- 保证旧书签可用
- 降低导航改造期间的回归风险
- 便于逐步替换代码内可能遗留的旧链接

## 4.3 页面文件命名建议

推荐使用新的 `Console*View.vue` 命名，而不是继续扩展 `Dashboard*View.vue`：

- `frontend/src/views/ConsoleAnalyticsTokensView.vue`
- `frontend/src/views/ConsoleAnalyticsOverviewView.vue`
- `frontend/src/views/ConsoleModelManagementView.vue`
- `frontend/src/views/ConsoleModelSelectionView.vue`

原文件处置建议：

- `DashboardAnalyticsView.vue`：拆分完成后删除，或保留为过渡壳并做 redirect/import 转发
- `DashboardConfigView.vue`：拆分完成后删除，或保留为过渡壳并跳转到模型管理页

推荐最终下线旧 view 文件，避免命名语义继续混乱。

## 5. Analytics 拆分方案

## 5.1 拆分原则

Analytics 的拆分不建议以“后端 API 拆分”为前提，而应基于现有 query 与图表组件做前端编排拆分。

核心原则：

- 共用同一套 filters
- 共用现有 overview / daily / events 数据源
- 拆分展示职责，而不是复制统计逻辑

## 5.2 页面职责划分

### 页面 A：Token 分析页

推荐包含：

#### 指标卡

- `Input Token`
- `Output Token`
- `单次总消耗`
- 可选保留 `Provider 覆盖率` 作为 token 准确度辅助说明

#### 图表

- `TokenBreakdownTrendChart`
- `TokenSourceShareChart`
- 可选增加 token 相关趋势说明卡

#### 可选附加

- 若想提升可解释性，可保留部分 token 相关 insight 文案
- 但不建议把请求、世界、事件类型分布继续放进该页

### 页面 B：综合分析页

推荐包含：

#### 指标卡

- `总请求`
- `成功率`
- `失败次数`
- `平均耗时`
- `设定命中`

#### 图表 / 面板

- `RequestsTrendChart`
- `LatencyTrendChart`
- `ModelDistributionChart`
- `WorldDistributionChart`
- `EventTypeDistributionChart`
- `AnalyticsEventsTable`
- `AnalyticsStoryMemoryPanel`
- `AnalyticsInsightPanel`

## 5.3 共享逻辑抽取建议

当前 `DashboardAnalyticsView.vue` 内已经有大量页面级 computed。拆分后如果直接复制，很快会出现：

- filters 重复
- worldLabelMap 重复
- distribution items 重复
- `refetchAll()` 重复

推荐新增一个页面级 composable，例如：

- `frontend/src/domains/analytics/composables/useAnalyticsDashboardState.ts`

建议其职责为：

1. 统一维护筛选条件
2. 暴露 header 所需 props
3. 暴露 overview / daily / events 查询结果
4. 暴露 metrics 的子集方法
5. 暴露各类 distribution items

这样两个页面只负责编排：

- Token 页只拿 token 子集
- 综合页只拿综合子集

## 5.4 过滤栏策略

推荐保留一个统一的 `AnalyticsHeader`，两页共用。

需要注意：

- 若 Token 页不展示事件表，则 `eventsLimit` 可不暴露在该页 UI 中
- 处理方式可以是：
  - 让 `AnalyticsHeader` 支持 feature flag
  - 或 Token 页复用 Header 的大部分能力，单独做一个轻量 wrapper

推荐方案：

- 不改原 `AnalyticsHeader` 的主体设计
- 增加可选 prop 控制是否显示 eventsLimit 相关输入

这样变更最小。

## 5.5 风险点

### 风险 1：两个页面重复请求相同数据

若两个页面都直接各自调用 overview/daily/events query：

- 用户切页可能看到重复请求
- 页面逻辑会重复

缓解：

- 先抽共享 composable
- 保持 query key 一致，让 TanStack Query 复用缓存

### 风险 2：指标口径在拆分中被意外改写

缓解：

- metrics 的原始计算逻辑不要重写
- 只做“展示子集”筛选

## 6. Config 拆分方案

## 6.1 页面职责划分

### 页面 A：模型管理页

该页面应保留 `DashboardConfigView.vue` 当前的大部分内容，不做产品层面的明显变化。

推荐保留：

- `ConfigPageHeader`
- `ProviderConfigCards`
- `GenerationParamsPanel`
- `ConfigResetPanel`（可保留）

其中最重要的约束是：

- “AI 提供商部分 + 生成参数部分”保持与当前版本几乎一致
- 不在这里混入三个场景的独立模型偏好

也就是说，模型管理页继续承担：

1. provider 连通性与凭据配置
2. provider 的模型列表查询
3. 默认模型 / 默认 provider 的管理
4. 生成参数配置

### 页面 B：大模型选择页

该页面新增，专门管理三个执行场景的模型偏好：

- 故事生成
- Prompt 优化
- 故事调整

页面应显示每个场景的：

- 场景说明
- 当前生效 provider
- 当前生效 model
- provider 切换
- model 切换
- 未配置时的回退说明

## 6.2 页面关系建议

推荐明确这两个页面的分工边界：

### 模型管理页负责

- provider 是否可用
- API Key 是否正确
- Base URL 是否正确
- provider 默认模型池管理
- 全局 generation params

### 大模型选择页负责

- 场景 A 用哪个 provider/model
- 场景 B 用哪个 provider/model
- 场景 C 用哪个 provider/model
- 场景配置的读取、保存、回显

这样可以避免用户认知混乱：

- “模型管理”是基础设施层
- “模型选择”是业务场景层

## 6.3 前端状态抽取建议

当前 `useDashboardConfig.ts` 同时管理：

- provider 流程
- model 列表
- defaultProvider/defaultModel
- generation params

拆分后不建议继续把所有逻辑都堆在一个 composable 里。

推荐拆为两层：

### 第一层：provider / management 逻辑

保留或演进当前：

- `useDashboardConfig.ts`

用于模型管理页。

### 第二层：scene model selection 逻辑

新增：

- `frontend/src/domains/settings/composables/useSceneModelSelection.ts`

负责：

- 拉取后端场景偏好
- 管理三个场景的 provider/model 表单状态
- 读取 provider 远程模型列表
- 保存场景偏好
- 计算回退提示文案

## 6.4 大模型选择页 UI 建议

推荐每个场景做成独立卡片：

- 故事生成
- Prompt 优化
- 故事调整

每张卡片包含：

1. 场景描述
2. provider 选择器
3. model 选择器
4. 获取模型按钮
5. 当前回退说明
6. 保存状态

### 回退文案建议

例如：

- 未单独配置时，将回退到当前默认模型 `deepseek-chat`
- 若已配置 provider 但 model 为空，则视为无效配置并提示用户完成选择

### 为什么不直接复用 ProviderConfigCards

`ProviderConfigCards` 的职责是 provider 基础设施管理，不是场景模型选择。

如果强行复用，会混入：

- API Key 输入框
- Base URL 输入框
- 测试按钮
- 默认 provider 徽标

这些都不适合出现在“按业务场景挑模型”的页面里。

因此建议：

- 复用 provider catalog / fetch models / selected model option 的底层能力
- 不直接复用 `ProviderConfigCards` 的整块 UI

## 7. 后端场景化模型偏好设计

## 7.1 设计原则

本期目标是“后端记录”，因此场景偏好必须以后端数据库为真相源。

不建议仅做：

- 前端 localStorage
- 前端 remote storage
- 继续把所有设置塞进 `user_config_v2`

因为这不满足“后端记录”的要求，也无法让多终端保持一致。

## 7.2 推荐数据结构

推荐继续使用现有 `user_settings` 表扩列，而不是新建独立表或 JSON blob。

### 推荐新增列

- `story_generation_provider`
- `story_generation_model`
- `input_enhancement_provider`
- `input_enhancement_model`
- `story_adjustment_provider`
- `story_adjustment_model`

### 推荐原因

1. 与当前 SQLite 使用方式一致。
2. 调试简单，直观可查。
3. 与 `UserSettings` 模型天然匹配。
4. 迁移成本低于引入独立配置表。
5. 便于后续继续扩展更多场景字段。

## 7.3 为什么不只存 model，不存 provider

当前系统里 provider 并不完全能从 model 绝对可靠推断，尤其在以下情况下：

- `custom` provider
- 同名模型跨 provider 暴露
- 用户通过自定义 base_url 代理不同后端模型

因此推荐场景化偏好同时存：

- provider
- model

`base_url` 不建议按场景重复存储，继续沿用现有 provider 级配置即可。

## 7.5 API 设计建议

推荐在现有 provider settings 体系下增加一个新的场景模型偏好接口，而不是新开与 provider 无关的 settings 路由。

### 推荐接口

#### 读取

`GET /api/v2/providers/scene-models`

#### 保存

`PUT /api/v2/providers/scene-models`

### 推荐响应结构

```json
{
  "story_generation": {
    "provider": "deepseek",
    "model": "deepseek-chat"
  },
  "input_enhancement": {
    "provider": "openai",
    "model": "gpt-4.1-mini"
  },
  "story_adjustment": {
    "provider": "qwen",
    "model": "qwen-max"
  },
  "fallback": {
    "default_model": "deepseek-chat"
  }
}
```

说明：

- `fallback.default_model` 用于提示当前后端侧兜底模型
- 若未来需要，也可以扩展 `resolved_provider` 辅助信息

### 保存结构建议

```json
{
  "story_generation": {
    "provider": "deepseek",
    "model": "deepseek-chat"
  },
  "input_enhancement": {
    "provider": "openai",
    "model": "gpt-4.1-mini"
  },
  "story_adjustment": {
    "provider": "qwen",
    "model": "qwen-max"
  }
}
```

### 为什么推荐 `scene-models` 而不是直接复用 `/providers/config`

`/providers/config` 当前语义非常明确：

- 保存 API Key
- 保存 Base URL

如果继续往这个接口塞业务场景偏好，会让 schema 语义变得混乱。

因此更合适的做法是：

- 依旧放在 `provider_routes.py` 中
- 但单独使用 `scene-models` 资源路径

## 8. 三个执行入口的接入方案

## 8.1 优先级规则

推荐统一使用以下解析优先级：

1. 请求显式传入的 provider/model
2. 后端记录的场景化 provider/model
3. 当前默认模型配置
4. 服务端 `settings.default_model`

这样做的好处：

- 保留显式 override 能力
- 老逻辑可兼容
- 场景配置真正成为默认业务选择，而不是硬覆盖所有请求

## 8.2 故事生成入口

涉及：

- `frontend/src/domains/story/composables/useStoryGeneration.ts`
- `story_rag_service/api/v2/story/generation_routes.py`
- `story_rag_service/services/story_generation/llm_factory.py`

### 前端改造要点

当前前端总是显式传 `defaultProvider/defaultModel`。

若要让后端场景偏好生效，必须调整为：

- 当用户没有本轮临时 override 时，不显式传 provider/model
- 让后端根据 `story_generation_*` 场景偏好决定

### 后端改造要点

在故事生成链路中引入场景解析：

- scene = `story_generation`

并在创建 LLM 前，解析场景偏好。

## 8.3 Prompt 优化入口

涉及：

- `previewEnhancedStoryInput(...)`
- `POST /api/v2/story/input-enhancement/preview`
- `StoryGenerator.preview_enhanced_input(...)`

### 场景映射

后端内部统一命名建议：

- `input_enhancement`

UI 文案仍然可以显示为：

- Prompt 优化
- AI 增强 Prompt

### 前端改造要点

和故事生成一样：

- 不再无条件显式传 `defaultProvider/defaultModel`
- 未指定时交由后端场景偏好解析

## 8.4 故事调整入口

涉及：

- `frontend/src/views/StoryAdjustmentView.vue`
- `story_rag_service/services/story_adjustment_service.py`

### 场景映射

- `story_adjustment`

### 前端改造要点

当前调整页直接读取 `configStore` 默认模型，这会覆盖新的场景偏好。

需要改为：

- 若页面没有显式“本次临时覆盖”需求，则不传 provider/model
- 由后端按 `story_adjustment_*` 场景偏好决定

## 8.5 Base URL 的处理原则

场景化模型偏好只负责 provider/model，不复制 base_url。

base_url 仍按：

- 当前 provider 对应的用户级 base_url 配置

这样能避免：

- 同一个 provider 在多个场景存多份重复 URL
- 修改 provider base_url 后，需要连带更新所有场景设置

## 13. 风险与应对

## 13.1 风险一：旧默认模型逻辑与场景偏好冲突

### 表现

前端若继续显式发送 `defaultProvider/defaultModel`，会导致后端场景偏好永远不生效。

### 应对

- 明确请求优先级
- 调整前端请求构造
- 补充联调验证用例

## 13.2 风险二：Prompt 优化的场景定义被误解

### 表现

开发时可能误以为要新增一个 Prompt 优化页面，偏离需求。

### 应对

- 在文档和接口字段中明确 `input_enhancement`
- UI 展示为“Prompt 优化”，内部实现映射到现有 input-enhancement preview

## 13.3 风险三：Analytics 拆分后产生重复逻辑

### 表现

两个新页面分别复制大量 computed / query / filter 逻辑。

### 应对

- 抽共享 composable
- 保持 query key 一致
- 页面只负责展示子集

## 13.4 风险四：数据库迁移兼容不足

### 表现

老用户数据库缺列导致读取失败。

### 应对

- 在 `services/database.py` 中沿用现有补列机制
- 启动后自动补齐新列
- 默认值全部允许为 NULL

## 13.5 风险五：控制台分组改造误伤现有管理页

### 表现

Lorebook / 剧本设计误被迁移或 sidebar 分组被整体重写。

### 应对

- 本期只移动 Config / Analytics 相关条目
- Lorebook / 剧本设计继续保留在“管理”中

## 15. 评审结论

### 15.1 推荐实施策略

推荐采用以下顺序：

1. 先做控制台分组与兼容路由
2. 再拆分 Analytics 页面
3. 再拆分 Config 页面，优先落模型管理页
4. 然后补后端场景化模型偏好
5. 再做大模型选择页
6. 最后切换三个执行入口到新偏好解析

这是风险最低、对现有页面侵入最小的顺序。

### 15.2 最终建议

本需求的关键不是“把两个页面拆成四个页面”本身，而是顺带把“单一默认模型”升级为“按业务场景选择模型”的能力。

因此，本方案的核心建议是：

- 前端页面拆分要与后端场景化模型偏好同步设计
- 但实施时分阶段推进，先做信息架构，再做偏好持久化，再接入消费链路
- 严格把“Prompt 优化”绑定到当前真实存在的 input enhancement 链路，避免范围漂移

如果按本方案执行，可以在不大改现有 provider 管理结构的前提下，完成：

- 控制台重组
- Analytics 双页拆分
- Config 双页拆分
- 三场景模型自由选择
- 后端记录与多终端一致性
