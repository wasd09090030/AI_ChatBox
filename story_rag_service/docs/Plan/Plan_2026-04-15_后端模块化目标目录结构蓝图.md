# Plan_2026-04-15_后端模块化目标目录结构蓝图

## 1. 文档定位

本文不是当前代码结构说明，而是为后续后端模块化改造准备的**目标蓝图**。

用途有三点：

1. 给后续重构提供稳定的目标目录结构参考。
2. 提前约束“哪些职责应该放在哪一层”。
3. 降低未来新增功能继续堆进 `service_context.py`、`story_generator.py`、`config.py` 的概率。

本文默认遵循以下前提：

1. 不做一次性推倒重来。
2. 对外 `/api/v2/*` 契约优先保持稳定。
3. 采用“渐进迁移 + 兼容桥接 + 每步可回滚”的实施方式。

---

## 2. 当前问题与本蓝图要解决的重点

结合仓库当前现状，后端还存在以下结构性问题：

1. `config.py` 虽然定义集中，但业务读取分散，应用层、服务层、图层都可能直接依赖全局 `settings`。
2. `api/service_context.py` 已经承担了组合根职责，但装配仍然高度集中在一个文件里，扩展时容易继续膨胀。
3. 路由层虽然已经比早期更干净，但很多接口仍通过 `ServiceContainer` 一次拿到整包依赖，而不是按用例注入。
4. `services/story_generator.py` 同时承担请求归一化、上下文拼装、LLM 调用、记忆更新、SSE 输出等多类职责，单文件过重。
5. `application/` 已经出现，但“应用层只依赖抽象端口”这一点还没有彻底落地。
6. 当前 `repositories/`、`services/`、`graph/` 的边界方向基本正确，但还缺少一个明确的目标目录蓝图来持续收口。

本蓝图重点解决的不是“怎么把目录改得更好看”，而是下面四件事：

1. 配置集中定义、集中解析、按领域分发。
2. 组合根显式化，依赖装配可分模块阅读。
3. 路由层面向用例，应用层面向端口，基础设施层面向实现。
4. 把故事生成主链路从“大编排器”拆为多个单一职责协作者。

---

## 3. 目标设计原则

### 3.1 分层原则

目标结构按五层组织：

1. `api/`：只做协议适配、请求校验、响应封装、依赖声明。
2. `application/`：组织跨服务用例、编排业务流程、定义输入输出 DTO、依赖抽象端口。
3. `domain/`：放稳定的领域规则、策略、纯业务对象。
4. `infrastructure/`：实现数据库、向量库、LLM、配置、可观测性等外部能力。
5. `bootstrap/`：作为组合根，负责把配置、实现和应用层装配起来。

### 3.2 依赖方向

依赖只允许单向流动：

1. `api -> application`
2. `application -> domain`
3. `application -> ports`
4. `infrastructure -> application/ports`
5. `bootstrap -> api/application/infrastructure`

明确禁止以下反向依赖：

1. `application` 直接导入 `config.settings`
2. `application` 直接依赖 `Sqlite*Repository`、`ChatOpenAI`、`Chroma`
3. `api` 直接调用具体仓储或自己拼底层依赖
4. `graph` 直接读取全局配置并自行决策业务规则

### 3.3 演进原则

目标蓝图不要求一次完成，建议分三步走：

1. 先引入 `bootstrap/`、`api/dependencies/`、`application/ports/`
2. 再拆 `story_generator.py` 和配置解析链路
3. 最后把 `services/` / `repositories/` 的实现逐步归入 `domain/` 与 `infrastructure/`

## 5. 每个新模块的职责定义

下面只解释**新增或需要重点引入的新模块**，避免和现有目录重复描述。

### 5.1 `bootstrap/`

#### `bootstrap/app_factory.py`

职责：

1. 创建 `FastAPI` 应用实例。
2. 注册 middleware、router、lifespan。
3. 统一把容器和请求级上下文挂到 `app.state`。

不负责：

1. new 具体业务服务。
2. 写业务逻辑。
3. 直接解析业务请求参数。

#### `bootstrap/container.py`

职责：

1. 定义应用容器对象。
2. 管理容器初始化、缓存、关闭时释放。
3. 作为 `bootstrap/modules/*` 的聚合承载体。

不负责：

1. 在一个文件里写完全部实例化细节。
2. 向路由暴露整包“万能服务访问”接口。

#### `bootstrap/config_resolver.py`

职责：

1. 把全局 `Settings` 转为领域化配置对象。
2. 明确配置优先级，例如“请求参数 > 用户配置 > 服务端默认值”。
3. 为 application 层提供稳定的配置输入，而不是让它直接依赖全局单例。

不负责：

1. 调用数据库。
2. 决定具体 HTTP 路由行为。

#### `bootstrap/dependency_registry.py`

职责：

1. 管理 API 层会用到的 provider 注册。
2. 提供 `get_generation_facade()`、`get_world_use_case()` 这类细粒度依赖入口。
3. 隐藏底层容器细节，减少路由层对容器的感知。

#### `bootstrap/modules/*`

职责：

1. 按业务域拆分装配逻辑。
2. 每个模块只装配自己负责的实现与用例。
3. 避免 `service_context.py` 式的单文件集中膨胀。

建议拆分：

1. `core.py`：数据库、日志、metrics、request 上下文。
2. `provider.py`：provider registry、user settings reader、LLM gateway。
3. `world.py`：world/story/lorebook 的 repository + application service。
4. `memory.py`：summary/runtime/entity/story_memory 读写相关。
5. `generation.py`：generation facade、request resolver、stream executor、post processor。
6. `script_design.py`：script design、roleplay、runtime 扩展能力。

### 5.2 `api/dependencies/`

#### `api/dependencies/common.py`

职责：

1. 提取 `request_id`、`user_id`、`thread_id` 等请求级上下文。
2. 统一容器访问方式。
3. 封装公共校验逻辑。

目的：

1. 让路由文件少写重复校验。
2. 避免每个路由直接操作 `app.state` 或容器。

#### `api/dependencies/generation.py`

职责：

1. 提供故事生成所需的 facade/provider。
2. 把流式和非流式共用的依赖获取方式统一起来。

#### `api/dependencies/memory.py`

职责：

1. 提供记忆查询与故事记忆快照用例。
2. 隔离 `memory_routes.py` 对具体 manager 的感知。

#### `api/dependencies/world.py`

职责：

1. 提供 world/story/lorebook 相关用例。
2. 让 `world_story_routes.py`、`lorebook_routes.py` 不再依赖整个容器。

### 5.3 `application/ports/`

这是依赖倒置落地的关键目录。

#### `repositories.py`

职责：

1. 定义 world/story/session/runtime/entity/script_design 等仓储端口。
2. 让 application 层只面向“能力”，不面向 `Sqlite*Repository` 实现类。

#### `llm_gateway.py`

职责：

1. 定义统一 LLM 调用抽象。
2. 屏蔽 OpenAI-compatible、Anthropic、DeepSeek 等差异。

#### `vector_store.py`

职责：

1. 定义检索、批量重建、按前缀删除等能力抽象。
2. 让 lorebook / retrieval 逻辑不直接依赖 Chroma 细节。

#### `user_settings.py`

职责：

1. 抽象用户级 provider/model/base_url/api_key 的读取能力。
2. 让 generation 配置决策不再直连 `user_manager`。

#### `feature_flags.py`

职责：

1. 抽象运行时开关，如 summary/entity/runtime/debug payload。
2. 避免 application 层直接判断全局 settings 字段。

#### `observability.py`

职责：

1. 抽象 metrics、analytics、审计事件。
2. 避免业务流程写死具体埋点实现。

### 5.4 `application/generation/`

这是后续拆 `story_generator.py` 的主落点。

#### `facade.py`

职责：

1. 提供统一的生成入口。
2. 对 API 层隐藏 graph / stream / preview 三种执行路径的内部差异。

#### `request_resolver.py`

职责：

1. 统一处理模型、provider、base_url、temperature、max_tokens、feature flags 的决议。
2. 统一处理用户显式参数、用户默认值、服务端默认值三层优先级。

#### `context_builder.py`

职责：

1. 负责会话上下文、memory bundle、world config、script guidance 等聚合输入。
2. 统一复用在 graph 和 stream 两条路径中。

#### `llm_executor.py`

职责：

1. 负责非流式生成执行。
2. 只处理“拿到已归一化请求后如何执行”，不负责请求默认值决议。

#### `stream_executor.py`

职责：

1. 负责流式执行和 chunk 级别事件协同。
2. 与路由层的 SSE 协议封装分离。

#### `post_processor.py`

职责：

1. 负责消息持久化、summary 更新、entity patch、story_memory 聚合等生成后处理。
2. 统一写路径，避免流式/非流式分别实现一份。

#### `response_builder.py`

职责：

1. 统一构造 v2 response、graph response、debug payload。
2. 避免路由层和 graph 节点重复组装响应对象。

### 5.5 `domain/`

`domain/` 只放稳定的业务规则，不放框架对象和外部实现。

#### `domain/generation/policies.py`

职责：

1. 放温度调整、choices 提取规则、token budget 策略、输入增强判定等纯规则。
2. 让这些规则不再散落在大编排器里。

#### `domain/memory/policies.py`

职责：

1. 放 story_memory 状态流转、summary stale/reset、timeline 聚合等规则。
2. 让记忆相关状态机成为可测试的纯逻辑。

### 5.6 `infrastructure/`

#### `infrastructure/persistence/`

职责：

1. 承载 SQLite 仓储实现。
2. 承载 JSON legacy 迁移兼容实现。
3. 明确“repository port 在 application，repository implementation 在 infrastructure”。

#### `infrastructure/llm/`

职责：

1. 承载 provider registry、base_url 决议、client factory。
2. 实现 `LLMGateway`。
3. 把当前 `services/story_generation/llm_factory.py` 的技术实现从业务流程里剥离出来。

#### `infrastructure/vector_store/`

职责：

1. 承载 Chroma 和 embedding 相关技术实现。
2. 对上暴露 vector store port，而不是直接暴露 Chroma client。

#### `infrastructure/config/`

职责：

1. 定义领域配置模型。
2. 负责把 `.env` / `settings` 转换为应用可消费的配置对象。

#### `infrastructure/observability/`

职责：

1. 承载 metrics、日志、埋点的技术实现。
2. 保证 observability 能力可替换、可扩展。

## 6. 现有目录到目标目录的迁移映射

为了避免后续实施时无从下手，下面给出推荐映射关系。

### 6.1 配置与装配

现状：

1. `config.py`
2. `api/service_context.py`
3. `main.py`

目标：

1. `config.py` 保留 `Settings`
2. 装配逻辑迁入 `bootstrap/`
3. `main.py` 仅作为启动薄入口

### 6.2 路由依赖

现状：

1. 路由广泛 `Depends(get_services)`
2. 路由可拿到整个 `ServiceContainer`

目标：

1. 路由改为依赖 `api/dependencies/*`
2. 每个路由只拿自己需要的用例或 facade

### 6.3 故事生成链路

现状：

1. `services/story_generator.py`
2. `application/memory/*`
3. `services/story_generation/*`
4. `graph/story_v2/*`

目标：

1. 流程编排上移到 `application/generation/*`
2. 纯规则下沉到 `domain/generation/*`
3. LLM/向量库技术实现下沉到 `infrastructure/*`
4. graph 只做图节点编排，不继续承载配置解析和大段业务逻辑

### 6.4 仓储实现

现状：

1. `repositories/*.py`

目标：

1. 端口定义放 `application/ports/repositories.py`
2. 具体实现放 `infrastructure/persistence/sqlite/`
3. `repositories/` 作为迁移期兼容目录逐步清空

## 7. 第一批建议创建的目录与文件

如果后续要开始真正实施，我建议第一批只创建下面这些“边界型文件”，不要一开始就大搬家：

```text
story_rag_service/
├── bootstrap/
│   ├── app_factory.py
│   ├── container.py
│   ├── config_resolver.py
│   └── modules/
│       ├── core.py
│       ├── memory.py
│       └── generation.py
├── api/dependencies/
│   ├── common.py
│   ├── generation.py
│   └── memory.py
├── application/ports/
│   ├── llm_gateway.py
│   ├── repositories.py
│   ├── user_settings.py
│   └── feature_flags.py
└── application/generation/
    ├── facade.py
    ├── request_resolver.py
    └── context_builder.py
```

原因：

1. 这批文件能先把边界立起来。
2. 迁移成本低，不需要立刻移动大量现有代码。
3. 一旦边界可用，就能逐步把旧逻辑挪进去，而不是继续向旧大文件加代码。

## 8. 落地顺序建议

### 阶段 A：先立边界，不大迁移

1. 新建 `bootstrap/`
2. 新建 `api/dependencies/`
3. 新建 `application/ports/`
4. 让新入口先和旧实现共存

### 阶段 B：收口配置与依赖

1. 收口 `settings` 直接读取
2. 建立配置解析器
3. 让路由层改用细粒度 provider

### 阶段 C：拆生成主链路

1. 抽出 `request_resolver`
2. 抽出 `context_builder`
3. 抽出 `post_processor`
4. 最后再瘦身 `story_generator.py`

### 阶段 D：仓储与基础设施归位

1. 定义 repository ports
2. 新增 infrastructure implementation
3. 逐步替换旧 `repositories/` 的直接依赖

---

## 9. 需要刻意避免的反模式

后续实施时，以下做法应明确避免：

1. 把 `bootstrap/modules/*` 再写成新的 `service_context.py`
2. 在 `api/dependencies/*` 里继续拼装业务流程
3. 在 `application/*` 里直接导入 `config.settings`
4. 在 `domain/*` 里依赖 FastAPI、LangChain、sqlite3、Chroma
5. 为了迁移方便把所有东西都先做成 Optional，导致新边界名义存在、实际失效
6. 只拆文件，不拆职责

---

## 10. 最终建议

本蓝图建议作为后续后端模块化的统一目标参考，核心判断标准不是“目录是否改成了新名字”，而是：

1. 配置是否真正集中解析。
2. 路由是否只依赖用例。
3. 应用层是否只依赖端口。
4. 基础设施实现是否被隔离。
5. 故事生成链路是否从单一大编排器拆成多协作者。

如果未来实施过程中需要在“理想分层”和“渐进迁移成本”之间取舍，优先保证三点：

1. 依赖方向正确。
2. 兼容现有主链路。
3. 每一步都能独立验证与回滚。
