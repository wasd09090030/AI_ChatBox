# Plan_2026-03-28_双模式故事创作技术设计方案

## 1. 背景与目标

本方案用于将当前仓库中的故事创作能力明确拆分为两种模式，并形成可落地的前后端技术设计：

1. 渐进式创作
2. 严格剧本创作

该设计不是推倒现有故事生成能力，而是在当前 `StoryView -> useStoryGeneration -> /api/v2/story/* -> StoryGenerator / Story Graph -> RAG / ScriptDesign / Summary Memory` 链路上做分层升级。

当前代码已经具备以下基础：

1. 前端已经支持故事请求携带 `script_design_id`、`active_stage_id`、`active_event_id`、`follow_script_design`、`selected_context_entry_ids` 等字段。
2. 后端 `StoryGenerationRequest` 已经具备剧本相关字段，且 `MemoryOrchestrator` 已将 `script_guidance` 与 `retrieved_lore` 分层放入 `MemoryBundle`。
3. 当前非流式与流式生成仍然以统一生成链路为核心，剧本设计在现阶段主要以 prompt 软约束参与，不属于真正的结构驱动执行。

本次方案的目标，是将上述“半分层”状态推进为“模式明确、边界清晰、可逐步迁移”的双模式架构。

### 1.1 目标

本次规划需要同时达成以下目标：

1. 明确区分两种创作模式的产品语义、前端交互和后端执行流。
2. 保留当前高自由度故事生成能力，作为渐进式创作主路径。
3. 为严格剧本创作建立独立的结构驱动执行流，使剧情推进主要由 `stage / event / foreshadow / runtime state` 决定，而不是由 prompt 主导。
4. 将 RAG 的职责收口为“背景知识检索”，在严格模式中不再承担主线驱动职责。
5. 引入独立的剧本运行态状态，用于承载当前剧情推进事实，而不是继续只依赖故事 metadata 和对话历史。
6. 保持旧请求、旧故事、旧前端页面在迁移期间可兼容运行。

### 1.2 非目标

本方案首期不做以下内容：

1. 不重写现有 `ScriptDesign` 模型本体；现有阶段、事件、伏笔模型继续作为规划层基础。
2. 不将剧本设计整体向量化并替代 Lorebook。
3. 不在首期引入复杂图形化时间线编辑器、多用户协作或自动剧情求解器。
4. 不在首期引入第二次 LLM 审核作为强依赖；一致性校验优先采用规则校验与轻量策略。
5. 不要求一次性替换全部流式与非流式接口；允许先建立兼容层再逐步收口。

### 1.3 核心原则

1. 渐进式创作是 `prompt 驱动 + RAG 参与 + 结构可选参考`。
2. 严格剧本创作是 `结构驱动 + prompt 仅做本轮微调 + RAG 只做背景补充`。
3. RAG、ScriptDesign、ScriptRuntime 必须是三个语义独立的层，而不是互相混写的字段集合。
4. 旧字段兼容优先，但新设计的真相源必须明确，避免长期双真相源并存。
5. 新能力应尽量复用现有模块，而不是绕开现有架构再建一套平行体系。

1. 严格模式一致性校验仍是轻量版，尚未覆盖更强的阶段跳跃、未授权实体、伏笔时机等规则。

4. 动态运行时 smoke test 本轮未完成，原因是当前 shell 的 `python3` 环境缺少后端依赖；已完成的验证是 Python 语法编译和前端生产构建。

本轮已完成的静态验证：

1. `python3 -m compileall story_rag_service` 通过。
2. `cd frontend && npm run build` 通过。
3. 构建过程中无新增 TypeScript 或 Python 语法错误。

## 2. 现状基线与代码事实

以下内容均基于当前仓库代码，不依赖假设。

### 2.1 前端故事创作链路现状

故事主页面位于 `frontend/src/views/StoryView.vue`，当前已经具备以下控制项：

1. 剧本选择：`selectedScriptDesignId`
2. 当前阶段：`selectedScriptStageId`
3. 当前事件：`selectedScriptEventId`
4. 是否跟随剧本：`followScriptDesign`
5. 关键角色对白控制：`selectedPrincipalCharacterId`、`dialogueMode` 等
6. Prompt 组合器：`selectedContextEntryIds`、focus template、authors note 等

请求拼装位于 `frontend/src/domains/story/composables/useStoryGeneration.ts`，当前发送故事生成时会固定带上：

1. `use_rag: true`
2. 可选的 `script_design_id / active_stage_id / active_event_id / follow_script_design`
3. 可选的 `selected_context_entry_ids`

这说明现有前端已经具备剧本辅助能力，但仍然以统一生成表单驱动，没有“模式级别”的执行分叉。

### 2.2 后端请求与生成现状

后端请求模型位于 `story_rag_service/models/story.py`。当前 `StoryGenerationRequest` 已包含：

1. RAG 控制字段：`use_rag`、`top_k`
2. 剧本字段：`script_design_id`、`active_stage_id`、`active_event_id`、`follow_script_design`
3. Prompt 与对话控制字段：`authors_note`、`mode`、`instruction`、`focus_instruction`、`dialogue_mode` 等

这说明后端请求模型已经足够承载“剧本辅助生成”，但尚未形成“模式级设计”。

### 2.3 Memory Bundle 现状

`story_rag_service/application/memory/orchestrator.py` 已将当前生成相关信息拆成多层：

1. `episodic`
2. `semantic`
3. `profile`
4. `procedural`
5. `world`

其中：

1. `procedural.script_guidance` 承载剧本提示信息。
2. `world.retrieved_lore` 承载 Lorebook / RAG 命中结果。

这说明系统已经具备“逻辑分层”，但在最终生成阶段仍然统一拼接进同一个 system prompt。

### 2.4 Prompt 组装现状

`story_rag_service/services/story_generation/prompt_builder.py` 中，当前剧本设计以 `ScriptDesignComponent` 形式注入 prompt。组件内容包括：

1. 当前剧本标题、主线概括、摘要、主题、核心冲突
2. 当前阶段目标、张力
3. 当前事件目标、阻碍、预期结果
4. 未回收伏笔提示

该实现的价值是“提供软约束”，但它仍属于 prompt 层，而不是结构执行层。

### 2.5 RAG 检索现状

`story_rag_service/application/story_generation/retrieval.py` 当前负责：

1. 基于用户输入与短期上下文构造检索 query
2. 规则命中
3. 向量检索
4. BM25 混合检索
5. 历史对话召回

当 `use_rag` 为 `false` 时会整体跳过 RAG。说明当前架构中的 RAG 仍然是“生成总入口的一部分”，没有按不同创作模式调整策略。

### 2.6 剧本设计现状

`story_rag_service/models/script_design.py` 已定义以下结构：

1. `ScriptDesign`
2. `ScriptStage`
3. `ScriptEventNode`
4. `ForeshadowRecord`
5. `ScriptGenerationPolicy`

这为严格剧本模式提供了天然的规划层基础，尤其是：

1. `linked_lorebook_entry_ids`
2. `participant_lorebook_entry_ids`
3. `prerequisite_event_ids`
4. `foreshadow_ids`

这些字段已经足够支撑首版“结构驱动 + 背景检索过滤”的方案。

### 2.7 当前问题总结

当前系统的主要问题不是“没有剧本能力”，而是：

1. 剧本设计与 RAG 检索虽然在逻辑上分层，但生成时仍然被统一当作 prompt 素材。
2. 没有独立的 `ScriptRuntime` 真相源，导致“当前阶段 / 当前事件 / 是否已完成”主要依赖前端传值或故事 metadata。
3. RAG 在所有模式中都扮演类似角色，无法体现“即兴模式”和“严格模式”的不同召回策略。
4. 用户虽然能选择剧本，但系统并不能保证生成严格服从阶段推进。

## 3. 目标架构总览

### 3.1 双模式定义

#### 3.1.1 渐进式创作

渐进式创作用于高自由度、强即兴、允许模型顺着当前输入和上下文自然生长的故事生成。

特点：

1. 用户 prompt 是主驱动力。
2. RAG 正常参与背景知识检索。
3. 剧本设计可不绑定；即使绑定，也只作为参考约束，不强制推进。
4. 适合开坑、试写、支线、角色互动、灵感探索。

#### 3.1.2 严格剧本创作

严格剧本创作用于长篇主线、商业化写作、需要稳定结构与节奏回收的故事生成。

特点：

1. 剧情推进服从 `stage / event / foreshadow / runtime state`。
2. Prompt 不负责主线决策，只负责本轮重点、风格和推进意图微调。
3. RAG 只做背景知识补充，不决定主线方向。
4. 生成结果需要经过结构一致性校验，再决定是否推进状态。

### 3.2 三层领域边界

为避免模式演进后再次混淆，本方案将故事创作明确拆成三层：

#### 3.2.1 RAG 背景知识层

职责：

1. 世界设定
2. Lorebook 条目
3. 角色背景
4. 地点百科
5. 世界历史事件与规则说明

不负责：

1. 决定主线推进到哪一步
2. 判定某个阶段是否完成
3. 替代 ScriptDesign 或 ScriptRuntime

#### 3.2.2 ScriptDesign 规划层

职责：

1. 定义故事蓝图
2. 定义阶段和事件节点
3. 定义伏笔与回收预期
4. 定义事件与世界知识的关联

不负责：

1. 记录某次具体故事实例已经推进到哪一步
2. 记录某个具体分支已触发的即时状态

#### 3.2.3 ScriptRuntime 运行态层

职责：

1. 记录某个故事实例当前推进到的阶段和事件
2. 记录已完成和已跳过事件
3. 记录已兑现与已废弃伏笔
4. 记录当前场景时间、地点、在场角色、运行态备注
5. 作为严格模式下的剧情真相源

不负责：

1. 替代世界知识存储
2. 替代剧本设计蓝图

### 3.3 目标执行原则

1. 渐进式模式优先复用现有生成链路。
2. 严格模式新增独立执行分支，但尽量复用现有 ScriptDesign、RAG、生成服务与前端页面容器。
3. 双模式共用同一个故事页面，但使用不同的控制面板密度和不同的后端执行策略。

## 4. 数据模型设计

### 4.1 故事请求模型扩展

建议在 `StoryGenerationRequest` 上新增以下字段：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| creation_mode | enum | 否 | `improv` / `scripted`，默认 `improv` |
| progress_intent | enum | 否 | `hold` / `advance` / `complete`，默认 `hold` |
| runtime_state_id | string | 否 | 当前故事对应的 ScriptRuntimeState id |
| strictness | enum | 否 | 预留字段，建议首期内部保留 `soft` / `strict` |
| allow_state_transition | bool | 否 | 是否允许本轮生成后自动推进状态 |

说明：

1. `creation_mode` 是本次设计的核心字段，不建议继续沿用 `follow_script_design` 作为模式开关。
2. `progress_intent` 用于表达用户本轮意图，不再只依赖自然语言 prompt 推测是否要推进或完成当前事件。
3. `follow_script_design` 在兼容期可保留，但不再作为模式真相源。

### 4.2 ScriptRuntimeState 模型

建议新增独立运行态模型：`ScriptRuntimeState`。

建议字段如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | string | 是 | 运行态状态 id |
| story_id | string | 是 | 绑定故事实例 |
| session_id | string | 是 | 绑定故事会话 |
| world_id | string | 是 | 所属世界 |
| script_design_id | string | 是 | 绑定剧本 |
| creation_mode | enum | 是 | `improv` / `scripted` |
| current_stage_id | string | 否 | 当前阶段 |
| current_event_id | string | 否 | 当前事件 |
| completed_event_ids | string[] | 否 | 已完成事件 |
| skipped_event_ids | string[] | 否 | 已跳过事件 |
| active_foreshadow_ids | string[] | 否 | 已埋设但未回收伏笔 |
| paid_off_foreshadow_ids | string[] | 否 | 已回收伏笔 |
| abandoned_foreshadow_ids | string[] | 否 | 已废弃伏笔 |
| current_location_entry_id | string | 否 | 当前地点条目 |
| current_time_label | string | 否 | 当前时间描述 |
| active_character_entry_ids | string[] | 否 | 当前在场关键角色 |
| runtime_notes | string | 否 | 运行态备注 |
| last_contract_snapshot | json | 否 | 最近一轮 round contract 快照 |
| last_check_result | json | 否 | 最近一轮一致性检查结果 |
| updated_at | datetime | 是 | 更新时间 |

说明：

1. 这是严格模式剧情推进的真相源。
2. 首期不要求替代现有 `StoryContext`；两者可以并存，但职责不同。
3. `StoryContext` 继续负责对话与短期上下文；`ScriptRuntimeState` 负责结构化剧情事实。

### 4.3 Round Contract 模型

严格模式生成前，系统需要先构建一份“本轮生成合同”。建议新增内部结构 `ScriptRoundContract`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| creation_mode | enum | 当前模式 |
| script_design_id | string | 剧本 id |
| stage_id | string | 当前阶段 |
| event_id | string | 当前事件 |
| stage_goal | string | 阶段目标 |
| event_objective | string | 本轮主要目标 |
| allowed_character_ids | string[] | 允许优先登场角色 |
| allowed_location_ids | string[] | 允许优先使用地点 |
| required_foreshadow_ids | string[] | 必须维护的伏笔 |
| blocked_event_ids | string[] | 当前不可越过的事件 |
| progress_intent | enum | 用户推进意图 |
| completion_guard | string | 何种条件下才允许判定完成 |
| rag_scope | json | 本轮允许检索的背景知识范围 |

说明：

1. 严格模式的核心不是“更多 prompt”，而是“先产出 contract，再去生成”。
2. Prompt 仅使用该 contract 的结果，而不是替代 contract。

### 4.4 一致性检查结果模型

建议为严格模式增加 `ScriptConsistencyCheckResult`：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| passed | bool | 是否通过 |
| stage_alignment | enum | `pass` / `warn` / `fail` |
| event_alignment | enum | `pass` / `warn` / `fail` |
| foreshadow_alignment | enum | `pass` / `warn` / `fail` |
| unauthorized_entities | string[] | 非授权重要实体 |
| premature_payoff_ids | string[] | 提前回收的伏笔 |
| unsupported_completion | bool | 未满足完成条件却试图完成 |
| notes | string[] | 校验说明 |

说明：

1. 首期建议采用规则校验。
2. 后续可扩展为 `规则校验 + LLM 审核` 的双层方案。

### 4.5 存储建议

建议将 `ScriptRuntimeState` 持久化到 SQLite，与现有运行时存储风格保持一致。

推荐原因：

1. 当前后端 runtime storage 已以 SQLite 为中心。
2. 剧情运行态需要稳定查询、更新与恢复，不适合仅存在前端状态或故事 metadata。
3. 严格模式后续可能需要按 `story_id / session_id / script_design_id` 多条件查询。

## 5. API 设计与兼容策略

### 5.1 故事生成接口扩展

现有以下接口保持存在：

1. `POST /api/v2/story/generate`
2. `POST /api/v2/story/generate/stream`
3. 相关 regenerate / rollback 接口

建议扩展请求载荷，但保持向后兼容：

新增字段：

1. `creation_mode`
2. `progress_intent`
3. `runtime_state_id`
4. `allow_state_transition`

兼容规则：

1. 未传 `creation_mode` 时默认视为 `improv`。
2. 旧字段 `follow_script_design = true` 时，若未显式传 `creation_mode`，可在兼容期映射为：
   - 默认仍走 `improv`
   - 但启用剧本参考提示
3. 只有当前端明确发送 `creation_mode = scripted` 时，才进入严格模式执行流。

### 5.2 运行态状态接口

建议新增一组运行态接口：

1. `GET /api/v2/story/runtime/{story_id}`
2. `PUT /api/v2/story/runtime/{story_id}`
3. `POST /api/v2/story/runtime/{story_id}/advance`
4. `POST /api/v2/story/runtime/{story_id}/complete`
5. `POST /api/v2/story/runtime/{story_id}/sync-from-metadata`（迁移期可选）

职责：

1. 查询当前故事的剧本运行态。
2. 显式更新当前阶段与当前事件。
3. 在用户明确选择“推进”或“完成”时，执行结构化状态迁移。

### 5.3 故事库 metadata 的兼容角色

当前前端会把以下字段写回故事 metadata：

1. `script_design_id`
2. `active_stage_id`
3. `active_event_id`
4. `follow_script_design`

兼容期建议如下：

1. 仍保留这些字段，以便旧故事和旧页面逻辑继续工作。
2. 但严格模式的真相源迁移到 `ScriptRuntimeState`。
3. metadata 在迁移期更适合扮演“浅绑定信息”和“兜底回填信息”，不再承担状态推进真相源角色。

## 6. 后端执行流设计

### 6.1 渐进式创作执行流

渐进式创作应尽量复用现有链路。

推荐执行顺序：

1. 读取会话与上下文。
2. 构建 `MemoryBundle`。
3. 执行现有 RAG 检索。
4. 可选加载剧本参考信息。
5. 使用现有 `StoryGenerator` 或 `Story Graph` 进行生成。
6. 正常写入历史、摘要、记忆事件。

特点：

1. 用户输入仍然是第一驱动。
2. 剧本设计如果存在，仅作为软提示。
3. 保留当前 `selected_context_entry_ids`、对白控制、authors note、focus instruction 的价值。

### 6.2 严格剧本创作执行流

严格模式需要独立执行分支，建议顺序如下：

1. 加载 `ScriptDesign`。
2. 加载或初始化 `ScriptRuntimeState`。
3. 解析当前 `current_stage_id / current_event_id`。
4. 结合 `progress_intent` 构建 `ScriptRoundContract`。
5. 根据 contract 决定本轮 RAG 检索范围。
6. 生成严格模式 system prompt：
   - 结构合同摘要
   - 用户本轮重点微调
   - 背景知识补充
7. 调用 LLM 生成正文。
8. 对输出执行结构一致性校验。
9. 若校验通过且允许推进，则更新 `ScriptRuntimeState`。
10. 记录校验结果、状态更新和激活日志。

### 6.3 严格模式与 Prompt 的关系

严格模式不等于完全不要 prompt，而是重新定义 prompt 的职责：

允许 prompt 控制：

1. 本轮叙事重点
2. 描写风格
3. 对话浓度
4. 节奏快慢
5. 是否尝试推进或完成当前事件

不允许 prompt 决定：

1. 当前主线推进到哪个阶段
2. 哪些关键事件被跳过
3. 哪些伏笔被提前回收
4. 哪些关键角色无条件加入主线

### 6.4 严格模式最小可行校验

首期建议只实现规则校验，避免过度设计。最低限度包括：

1. 若本轮未满足 `completion_guard`，禁止把事件标记为完成。
2. 若输出引入 contract 未授权的重要角色或地点，标记 `warn` 或 `fail`。
3. 若输出出现与当前阶段明显不符的事件跳跃，标记 `fail`。
4. 若输出提前回收未到时机的伏笔，标记 `warn` 或 `fail`。

### 6.5 流式与非流式差异

当前流式与非流式路径都存在。迁移策略建议：

1. 先在非流式路径完成严格模式的最小实现，便于调试 contract、校验和状态推进。
2. 流式路径首期可先支持：
   - 严格模式生成
   - 但状态推进延后到最终 done 事件后处理
3. 待非流式路径稳定后，再收紧流式路径的一致性行为。

## 7. RAG 策略设计

### 7.1 渐进式模式下的 RAG

渐进式模式保留现有 RAG 行为：

1. 正常根据用户输入、地点、活跃角色构造 query。
2. 正常执行规则命中、向量检索、BM25、历史检索。
3. 允许从世界知识中获得意外灵感与支线材料。

### 7.2 严格模式下的 RAG

严格模式中，RAG 必须收口为背景知识补充，不得成为剧情方向的主导因素。

推荐策略：

1. 先根据 `current_stage_id / current_event_id` 解析出关联实体。
2. 关联实体主要来自：
   - `linked_lorebook_entry_ids`
   - `participant_lorebook_entry_ids`
   - `linked_role_ids`
3. 再据此构建 `rag_scope`。
4. 检索时优先只允许访问该 scope 下的知识条目。

### 7.3 严格模式下的检索目标

严格模式中的 RAG 应只回答以下问题：

1. 当前地点有哪些世界规则或历史背景？
2. 当前角色的设定细节是什么？
3. 当前事件涉及的组织、道具、地点有哪些既有设定？
4. 当前阶段需要引用哪些历史事实或世界规则？

严格模式中的 RAG 不应该回答：

1. 主角下一步应该去哪条主线
2. 当前事件是否已经完成
3. 是否应该切换到下一个高潮节点

### 7.4 向量存储边界建议

本方案建议继续使用 Lorebook / World 作为向量存储的主要来源，但不建议将 ScriptDesign 直接作为普通向量条目并入主检索空间。

推荐做法：

1. RAG 数据源保留世界、Lorebook、关键背景条目。
2. ScriptDesign 通过结构化加载进入 runtime contract，不走自由检索。
3. 若未来需要“剧本检索”，也应作为单独索引或单独数据通道，而不是混入背景知识通道。

## 8. 前端交互设计

### 8.1 页面总体原则

当前实现采用“两条独立路由 + 一套共享工作台内核”的方式：

1. `/story/improv`：渐进式创作
2. `/story/scripted`：严格剧本创作

两条路由都复用 `StoryView.vue`，通过 `pageMode` 锁定模式，并在文案、控制入口和侧边栏上体现不同的工作方式。

这样做的原因是：

1. 用户层面形成明确入口，避免在同一个页面里反复切换模式。
2. 工程层面仍共享故事库、生成链路与大部分页面结构，避免复制两套创作逻辑。
3. 后续可以继续拉大两条页面的交互差异，而不破坏底层复用。

### 8.2 渐进式创作交互

渐进式模式保留当前主要交互：

1. 用户自由输入
2. Prompt Composer
3. Focus template
4. Authors note
5. 对话模式与关键角色对白控制
6. 可选的剧本参考绑定

特点：

1. 输入框是主中心。
2. 剧本相关控件允许折叠到较轻量位置。

### 8.3 严格剧本创作交互

严格模式需要显著突出“剧本主导”和“Prompt 仅做微调”。

当前实现已经落地以下结构：

1. 严格剧本创作为独立路由页面。
2. 页头保留模式徽标与跨模式跳转入口。
3. 底部 Prompt 区上方改为稳定双工具区：
   - 左侧固定“剧本推进”
   - 右侧固定“本轮微调说明”
4. “剧本推进”按钮打开独立 `StoryScriptSidebar`，集中控制剧本、阶段、事件和推进操作。
5. “本轮微调说明”明确告知用户：这里只调整本轮表现，不负责主线调度。

严格模式需要在 `StoryScriptSidebar` 或相关侧栏中呈现：

1. 当前剧本
2. 当前阶段
3. 当前事件
4. 阶段目标
5. 事件目标与阻碍
6. 当前需维护的伏笔
7. 用户本轮推进意图

推荐新增结构化控件：

1. 创作模式选择：`渐进式` / `严格剧本`
2. 进度意图：`仅描写` / `推进事件` / `完成事件`
3. 本轮重点输入框：用于细调本轮叙事焦点
4. 状态推进结果展示：本轮是否完成事件、是否有一致性告警

当前完成情况：

1. 已完成独立侧边栏中的剧本、阶段、事件选择器。
2. 已完成“保存当前推进 / 推进到下一事件 / 推进到下一阶段”按钮。
3. 已完成严格模式底部工具区的语义重命名与布局稳定化。
4. 已完成严格模式角色控制抽屉收口，只保留主角人设与关键角色交流配置。
5. 已完成严格模式 `?` 帮助内容替换为“微调说明”，强化“prompt 只做本轮微调”的产品语义。
6. 已完成渐进式模式剧本设计区的手风琴式折叠，降低非严格模式下的控制面板占用。
7. 已完成严格模式撤销的持久化闭环，撤销后 story 段落删除会同步写回仓库。
8. 已完成严格模式结构态撤销：撤销最后一段时会按上一段 `runtime_state_snapshot` 回退阶段 / 事件 / metadata，而不是只删除文本。
9. 已为新生成的严格模式故事补充 `runtime_initial_snapshot` 基线，支持撤销到故事开头时恢复初始运行态。
10. 尚未完成更丰富的运行态结果展示，例如 contract 预览、检查结果说明、推进后状态对比。

### 8.4 前端状态建议

建议在 `useStoryGeneration` 周边新增以下状态：

1. `creationMode`
2. `progressIntent`
3. `runtimeState`
4. `roundContractPreview`
5. `consistencyWarnings`

其中：

1. `creationMode` 和 `progressIntent` 直接参与请求拼装。
2. `runtimeState` 应通过后端接口拉取，而不是只在前端本地构造。

当前完成情况补充：

1. 已新增 `frontend/src/stores/storyDraftState.ts`，把前端草稿状态拆成两层：
   - 故事级共享草稿：跨 `/story/improv` 与 `/story/scripted` 同步
   - 路由级草稿：仅在当前 `pageMode` 下保留
2. 当前已明确同步的状态：
   - `selectedPersonaId`

   - `dialogueMode / dialogueTarget / dialogueIntent / dialogueStyleHint`

   - `selectedFocusTemplateId`
3. 当前已明确隔离的状态：
   - `userInput`
   - 渐进式创作页的 `storyMode / authorsNote / instruction`

6. 这一步的目标不是让两个页面“完全共享所有本地状态”，而是把“故事级真相”和“页面级临时草稿”分开，避免 KeepAlive 下出现双真相源。

### 8.5 旧控件迁移建议

现有控件的迁移建议如下：

1. `followScriptDesign`
   - 兼容期保留
   - 后续弱化为渐进式模式中的“参考剧本提示”
2. `selectedScriptStageId / selectedScriptEventId`
   - 兼容期保留
   - 严格模式下应以 `runtimeState.current_stage_id / current_event_id` 为主
3. `selectedContextEntryIds`
   - 渐进式模式继续保留
   - 严格模式下可作为附加背景上下文，但不覆盖 contract

补充说明：

1. 严格模式当前已经不再把“大块剧本推进面板”放在正文主内容区，而是迁移为可开关的右侧专用侧边栏。
2. 原“设定 / 记忆”侧边栏中的剧本推进区块，在严格模式路由下已隐藏，避免和专用剧本推进侧边栏重复。

## 11. 风险与对策

### 11.1 双真相源风险

风险：

1. metadata、前端状态、runtime state 同时记录阶段事件，容易冲突。

对策：

1. 兼容期明确优先级：
   - `scripted` 模式下以 runtime state 为准
   - `improv` 模式下 metadata 仅作参考

### 11.2 严格模式过死风险

风险：

1. 如果 contract 过硬，故事可能变得机械，失去文学性。

对策：

1. 只把“主线推进与结构边界”设为硬约束。
2. 把文风、镜头、对话浓度保留给 prompt 微调。
3. `progress_intent = hold` 时允许更多描写自由度。

### 11.3 RAG 误召回风险

风险：

1. 即使进入严格模式，如果 RAG scope 不够窄，仍可能把剧情带偏。

对策：

1. 严格模式优先做基于 `event/stage` 的过滤检索。
2. 激活日志中明确区分“背景补充”与“主线结构”来源。

### 11.4 流式与非流式行为分裂风险

风险：

1. 非流式先实现严格校验后，流式若仍然走旧逻辑，用户体验会分裂。

对策：

1. 先把严格模式能力落在非流式上。
2. 明确流式模式的过渡期限制，并尽快补齐 done 阶段处理。

### 11.5 旧故事绑定风险

风险：

1. 已存在的故事可能只有 metadata，没有 runtime state。

对策：

1. 提供 runtime state 初始化逻辑。
2. 若未找到 runtime state，则根据 metadata 和 script design 生成初始状态。

---

## 12. 验证与验收要点

### 12.1 渐进式模式验收

需要验证：

1. 不绑定剧本时，现有即兴故事功能无回归。
2. 绑定剧本但不进入严格模式时，仍能作为参考提示参与生成。
3. RAG、角色对白控制、focus prompt、authors note 等旧能力可继续使用。

### 12.2 严格模式验收

需要验证：

1. 进入严格模式后，生成请求会正确读取 runtime state。
2. 本轮输出重点可以通过结构化字段和微调 prompt 改变。
3. 未满足条件时不会错误推进事件或阶段。
4. 伏笔不会被无故提前回收。
5. RAG 命中主要来源于当前 stage/event 的关联知识，而不是全局自由漂移。

### 12.3 兼容验收

需要验证：

1. 老版本前端请求仍能正常生成。
2. 老故事仍能打开、重生成、回滚。
3. 不带 `creation_mode` 的请求默认按旧体验运行。

---

## 13. 推荐实施顺序

推荐优先级如下：

1. 先补请求字段与类型定义
2. 再落地 runtime state
3. 再做严格模式非流式执行流
4. 再做过滤式 RAG 与规则校验
5. 最后做流式对齐和前端强化

原因：

1. 这样能最大程度复用现有系统，最小化回归面。
2. 可以尽早让“严格模式”形成最小可用闭环，而不是先在 UI 上堆控制项。

补充说明：

1. 截至当前，这个推荐顺序已经基本按计划执行到“阶段 4 部分完成、阶段 5 部分完成”的位置。
2. 后续继续实施时，优先级建议调整为：
   - 强化严格模式一致性校验
   - 补前端运行态结果、contract 预览与告警可视化
   - 评估是否增加显式 `advance/complete` 专用 API

---

## 14. 结论

本方案的核心不是增加更多 prompt 开关，而是把当前故事创作正式升级为两条语义清晰的模式：

1. 渐进式创作：保留当前优势，继续服务即兴探索与自由创作。
2. 严格剧本创作：建立结构驱动、状态可追踪、RAG 收口的主线创作能力。

从仓库现状看，这一改造具备较好的基础条件：

1. 前端已经具备剧本相关控制字段。
2. 后端请求模型和 MemoryBundle 已经完成部分分层。
3. ScriptDesign 结构已经足够支撑首版严格模式。

因此，本方案推荐采用“兼容旧链路、增量引入新真相源、逐步收紧严格模式”的演进策略，而不是一次性重写故事生成系统。
