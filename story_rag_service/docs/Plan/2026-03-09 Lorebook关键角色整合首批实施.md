# 2026-03-09 Lorebook关键角色整合首批实施

## 本次目标

本次改动聚焦四件事：

1. 定义“关键角色交流”首批协议字段，并让字段从前端故事页传到后端生成链路。
2. 在 Lorebook 角色模型中补齐关键角色所需的基础字段，作为后续迁移落点。
3. 在故事创作控制抽屉中加入“关键角色交流”控制区，替代独立角色对话页的第一阶段能力。
4. 提供一份可直接运行的 smoke API 验证脚本，校验新字段已被 API 正确接收。

## 字段设计

### 故事生成协议字段

新增字段：

| 字段 | 类型 | 含义 | 首批用途 |
|---|---|---|---|
| `principal_character_id` | `string?` | 当前轮次的关键角色条目 ID | 将该角色显式提升为本轮核心交流对象 |
| `dialogue_mode` | `auto \| focused \| required` | 白话约束模式 | `auto` 自动融入，`focused` 优先发言，`required` 本轮必须出现对白 |
| `dialogue_target` | `string?` | 对话目标 | 主角、某角色、某群体 |
| `dialogue_intent` | `string?` | 对话目的 | 例如施压、透露情报、缓和关系、制造冲突 |
| `dialogue_style_hint` | `string?` | 风格提示 | 例如冷淡、尖锐、克制、诱导式 |
| `force_dialogue_round` | `boolean` | 是否强制本轮出现明确对白 | 用于和 `required` 模式保持显式一致 |

### Lorebook 角色字段

本次先在角色模型中显式加入这些字段，避免关键角色能力全部躲在松散 metadata 里：

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---|---|
| `role_tier` | `npc \| principal` | `npc` | 区分普通 NPC 与关键角色 |
| `dialogue_enabled` | `boolean` | `false` | 是否允许作为重点对白对象 |
| `opening_line` | `string?` | `null` | 建议开场白 |
| `example_dialogues` | `string[]` | `[]` | 多条对白示例 |
| `story_function` | `string?` | `null` | 剧情定位，如线索源、对手、导师、关系锚点 |

## UI 结构草案

### 1. 故事创作页

位置：故事创作抽屉中的“创作控制”下方。

结构：

1. 关键角色选择器
2. 对白模式选择器
3. 对话目标输入框
4. 交流意图输入框
5. 风格提示文本域

设计原则：

1. 不做独立 mini chat 消息流，避免再次复制一个角色对话产品。
2. 该面板只服务当前轮故事生成，是“叙事控制器”，不是“聊天页搬家”。
3. 当用户选择关键角色时，系统自动把该角色条目并入 `selected_context_entry_ids`，保证显式设定优先。

### 2. Lorebook 管理页

本次先完成基础承载：

1. 角色编辑表单新增关键角色字段。
2. 角色卡片列表新增“关键角色”“对话特化”徽标。
3. 暂未拆分出“普通 NPC / 关键角色”双筛选视图，这是下一批 UI 重构目标。

## 迁移映射表

旧独立角色卡迁移到 Lorebook 关键角色时，推荐使用以下映射：

| 旧角色卡字段 | 新 Lorebook 字段 | 说明 |
|---|---|---|
| `name` | `name` | 直接映射 |
| `description` | `background` 或 `description` 主体 | 根据内容语义拆分，首迁可全部落入 `background` |
| `system_prompt` | `story_function` + 后续 metadata 备份 | 不建议原样塞入提示词；先保留摘要用途 |
| `first_message` | `opening_line` | 作为建议开场白 |
| `example_messages[]` | `example_dialogues[]` | 一对一迁移 |
| `tags[]` | `keywords[]` 或 metadata tags | 优先转关键词，保留可检索性 |
| `metadata` | metadata | 原样备份，供后续清洗 |
| 无 | `role_tier='principal'` | 迁移后的旧角色卡默认为关键角色 |
| 无 | `dialogue_enabled=true` | 迁移后的旧角色卡默认开启对白特化 |

验证方式：

2. monkey patch `run_story_graph`，避免依赖完整运行时和模型调用。

4. 校验返回体包含对应 activation log，证明 API 合约已接通。
