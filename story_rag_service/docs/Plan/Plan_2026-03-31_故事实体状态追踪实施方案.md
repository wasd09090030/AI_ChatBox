# Plan_2026-03-31_故事实体状态追踪实施方案

## 1. 背景与目标

当前仓库中的故事创作系统，已经具备以下几类可持久化能力：

1. 会话消息真相源
2. 摘要记忆 `summary_memory`
3. 轻量剧情状态 `story_state`
4. 严格剧本运行态 `runtime_state`
5. 记忆更新事件日志 `memory_update_journal`

这些能力足以支撑“故事能继续写、摘要能保留、回滚可对账”的第一阶段需求，但还不足以支撑“每个主要人物实体的状态可追踪、可回放、可回滚、可前端展示”的目标。

当前系统中，人物实体的地点、携带物、受伤状态、同行关系、即时目标等信息，主要仍通过以下方式间接维持：

1. 模型根据上下文自行保持一致
2. `summary_text` 和 `key_facts` 做叙事压缩
3. 剧本模式通过 `runtime_state` 约束主线推进
4. Prompt 中使用“保持状态连续”的软约束

这意味着系统在“叙事连续性”上已经有一定能力，但在“结构化实体事实”上仍缺少独立真相源。

本方案的目标，是在当前架构基础上，增量引入一套“故事实体状态追踪系统”，使主要人物实体至少能够稳定追踪以下信息：

1. 当前所处地点
2. 当前状态标签，如受伤、警觉、疲惫、昏迷
3. 当前携带物
4. 当前同行或在场关系
5. 当前短期目标或状态说明
6. 状态变化证据与最近更新时间

并同时满足以下工程目标：

1. 不替换现有 `summary_memory`、`story_state`、`runtime_state`
2. 优先复用现有 SQLite、`memory_updates`、`StoryGenerator`、`StoryView`
3. 支持生成后更新
4. 支持 `rollback / regenerate / story adjustment commit` 后的重建
5. 支持前端当前状态展示和最近变更审计
6. 支持灰度开启与失败降级

## 2. 现状基线与代码事实

以下结论均基于当前仓库代码，而不是推测。

### 2.1 会话消息是当前故事上下文的持久化真相源

- `story_rag_service/services/session_manager.py`
  - `story_sessions` 保存会话元数据
  - `story_session_messages` 保存历史消息
  - 内存中的 `StoryContext.messages` 只是热缓存

这说明：故事继续生成时，真正可靠的底层事实仍然是消息真相源，而不是前端缓存。

### 2.2 摘要记忆已持久化，但不适合作为实体状态真相源

- `story_rag_service/services/summary_memory_manager.py`
  - `conversation_summaries` 表保存 `summary_text`、`key_facts`、`entities`
- `story_rag_service/services/story_generation/summary_helpers.py`
  - 当前同步路径 `build_summary_snapshot(...)` 返回 `entities: {}`
  - 默认只基于最近 3 条用户消息和 3 条助手消息构造快照

这说明：

1. 当前生产链路下，`entities` 并不稳定产生
2. 即使未来走 LLM 摘要，也更偏向“名字列表”而不是“结构化状态”
3. 摘要会截断、合并、压缩，不适合承载细粒度可回滚事实

### 2.3 RP 的 `story_state` 只适合剧情抽象状态，不适合实体世界态

- `story_rag_service/models/roleplay.py`
  - `StoryState` 包含 `chapter / objective / conflict / clues / relationship_arcs`
- `story_rag_service/graph/story_v2/nodes.py`
  - `update_story_state_node(...)` 在生成前执行
  - 更新依据主要是用户输入，而不是本轮最终生成结果

这说明 `story_state` 更适合“主线目标和线索状态”，不适合记录“张三现在在钟楼顶层，身上有铜钥匙”这类事实。

### 2.4 严格剧本 `runtime_state` 已有结构化字段，但尚未自动维护人物世界态

- `story_rag_service/models/story_runtime.py`
  - 已包含 `current_location_entry_id`
  - 已包含 `active_character_entry_ids`
  - 已包含 `current_time_label`
- `story_rag_service/services/story_runtime_manager.py`
  - 自动推进逻辑主要围绕 `stage / event / foreshadow`
  - 尚未看到生成后自动解析并写回地点或角色在场信息

这说明：

1. `runtime_state` 是主线结构态真相源
2. 它已经预留了部分场景态字段
3. 但还不是逐实体状态真相源

### 2.5 当前后端已经具备统一 post-generation 更新入口

- `story_rag_service/services/story_generator.py`
  - 非流式 `generate_story(...)`
  - 同步 `generate_story_sync(...)`
  - 流式 `generate_story_stream(...)`
  - 三条路径最终都会调用 `MemoryUpdateService.run_post_generation_updates(...)`

这说明：新增实体状态抽取与持久化，最合适的接入位置不是 route，而是 `StoryGenerator` 的生成后处理区。

### 2.6 记忆更新日志已具备稳定审计基础设施

- `story_rag_service/application/memory/journal.py`
  - `memory_update_journal` 已存 `event_id / operation_id / sequence / before / after / status`
- `story_rag_service/api/v2/memory_routes.py`
  - 已有会话级时间线查询接口

这说明：实体状态变更如果沿用 `memory_updates` 体系，可以直接继承现有审计和前端展示能力。

### 2.7 前端故事页已经能消费生成后的快照与事件

- `frontend/src/domains/story/composables/useStoryGeneration.ts`
  - 已消费 `summary_memory_snapshot`
  - 已消费 `memory_updates`
  - 已消费 `runtime_state_snapshot`
- `frontend/src/stores/storySession.ts`
  - 已按 session 持久化摘要和最近事件
- `frontend/src/components/story/StoryMemorySidebar.vue`
  - 已有记忆面板与最近事件展示区

这说明：前端增加实体状态面板的最佳路径，是在现有故事页与管理页记忆面板上增量扩展，而不是新开一套完全独立页面。

### 2.8 Lorebook 角色模型有动态字段，但当前没有自动回写闭环

- `story_rag_service/models/lorebook.py`
  - `Character` 已包含 `inventory`、`current_location`
- `story_rag_service/api/v2/lorebook_routes.py`
  - 当前仅存在通用 `update_lorebook_entry(...)` 手动更新入口

这说明：Lorebook 角色可以作为实体的静态设定源或初始值来源，但不适合作为首期“逐回合动态状态真相源”。

## 3. 问题定义

当前系统的核心缺口不是“没有状态”，而是“没有统一、结构化、逐实体、可回放的动态事实层”。

具体表现为：

1. 实体事实分散在消息、摘要、Prompt 软约束和少量 runtime 字段中
2. 同一个人物的地点、道具、状态无法以结构化形式稳定读取
3. rollback / regenerate 只能确保消息、摘要、向量索引趋于一致，无法保证人物动态状态同步回滚
4. 前端只能看到摘要和事件，无法直接回答“谁此刻在哪里、发生了什么变化”

因此，本次设计需要补齐的是：

1. 实体状态真相源
2. 实体状态变更事件
3. 实体状态重建机制
4. 实体状态前端展示与审计

## 4. 设计目标与非目标

### 4.1 设计目标

1. 为每个主要人物实体建立当前状态快照
2. 为每轮状态变化建立可回放 patch 事件
3. 支持以 `story_id + entity_id` 为中心读取当前状态
4. 支持从 persisted messages 或 story segments 重建实体状态
5. 让前端可展示“当前状态 + 最近变更 + 证据”
6. 让新系统与现有 `summary_memory`、`story_state`、`runtime_state` 边界清晰

### 4.2 非目标

1. 首期不做完整知识图谱
2. 首期不做非人物实体的全量覆盖
3. 首期不做复杂人工状态编辑器
4. 首期不做多模型抽取编排与成本优化系统
5. 首期不要求把动态状态写回 Lorebook 主数据

---

## 5. 核心设计原则

### 5.1 独立真相源原则

实体状态必须是独立事实层，而不是继续附着在 `summary_text` 上。

### 5.2 Patch 合并原则

每轮更新只写“明确变化”，不允许模型全量重写整个角色档案。

### 5.3 证据优先原则

任何结构化状态都应保留文本证据或证据摘要，便于审计与前端解释。

### 5.4 可重建优先原则

`current state` 必须可由事件流或消息真相源重建，而不是只依赖一次性写入结果。

### 5.5 最小侵入原则

优先复用：

1. SQLite
2. `StoryGenerator` 的 post-generation 更新入口
3. `memory_updates` / `memory_update_journal`
4. `StoryView` / `StoryMemorySidebar` / dashboard memory 页面

### 5.6 边界清晰原则

四类状态职责必须保持清楚：

1. `summary_memory`：压缩叙事记忆
2. `story_state`：剧情抽象状态
3. `runtime_state`：主线结构态和场景态
4. `entity_state`：逐实体动态事实层

## 6. 目标架构

### 6.1 新增一层 Entity State

建议在现有记忆架构旁新增：

1. `entity_state_current`
2. `entity_state_events`

其中：

- `entity_state_current` 用于快速读取“当前事实”
- `entity_state_events` 用于回放、对账、重建

### 6.2 Entity State 在整体链路中的位置

目标链路如下：

1. 读取 session messages / summary / runtime / story_state / lorebook
2. 生成故事正文
3. 生成后执行：
   - 原始消息写入
   - history index 更新
   - summary 更新
   - entity state 抽取与合并
4. 返回：
   - `memory_updates`
   - `summary_memory_snapshot`
   - `runtime_state_snapshot`
   - `entity_state_snapshot`
   - 可选 `entity_state_updates`

### 6.3 与现有三类状态的边界

#### 6.3.1 与 `summary_memory` 的关系

`summary_memory` 继续负责：

1. 长对话压缩
2. 关键事实提炼
3. 少量实体名回顾

`entity_state` 负责：

1. 逐实体结构化事实
2. 当前状态读取
3. 最近变更回放

摘要不再承担实体状态真相源角色。

#### 6.3.2 与 `story_state` 的关系

`story_state` 继续负责：

1. 当前章节
2. 当前目标
3. 主要冲突
4. 线索与关系弧的抽象描述

`entity_state` 不负责替代这些剧情抽象层，而是记录“谁在何处、有什么、状态怎样”。

#### 6.3.3 与 `runtime_state` 的关系

`runtime_state` 继续负责：

1. 当前 stage / event
2. 已完成与待推进结构
3. 当前场景级 location / active characters 的主线语义

`entity_state` 负责补充并细化：

1. 每个角色自己的地点
2. 每个角色的随身物与状态标签
3. 角色级 evidence 和最近变化

二者关系应为：

- `runtime_state` 是场景级、结构级真相
- `entity_state` 是角色级、事实级真相

## 7. 数据模型设计

### 7.1 实体标识策略

MVP 使用以下主键策略：

1. `story_id`
2. `session_id`
3. `entity_id`
4. `entity_type`

其中：

- `entity_id` 优先使用 Lorebook entry id
- `entity_type` 首期固定为 `character`
- `display_name` 作为展示字段，不参与唯一性判断

如果文本里出现未绑定 Lorebook 的临时人物：

1. MVP 不自动纳入 facts 层
2. 可只作为 evidence 或候选实体记录到调试层
3. 二期再考虑临时实体分配策略

### 7.2 `entity_state_current` 建议字段

建议字段：

1. `story_id`
2. `session_id`
3. `entity_id`
4. `entity_type`
5. `display_name`
6. `current_location`
7. `inventory`
8. `status_tags`
9. `companions`
10. `short_goal`
11. `state_summary`
12. `evidence`
13. `last_source_turn`
14. `updated_at`
15. `version`
16. `metadata`

说明：

- `current_location`：字符串或 location entry id，MVP 可先存名称，后续再提升为 entry id
- `inventory`：字符串数组
- `status_tags`：字符串数组，如 `受伤`、`警觉`
- `companions`：entity id 或角色名数组
- `short_goal`：短期目标
- `state_summary`：给前端和 prompt 使用的短摘要
- `evidence`：最近一次明确证据列表

### 7.3 `entity_state_events` 建议字段

建议字段：

1. `event_id`
2. `story_id`
3. `session_id`
4. `entity_id`
5. `entity_type`
6. `operation_id`
7. `sequence`
8. `source`
9. `source_turn`
10. `action`
11. `before_patch`
12. `after_patch`
13. `evidence`
14. `status`
15. `committed_at`

其中 `action` 建议包括：

1. `created`
2. `updated`
3. `rebuilt`
4. `reset`
5. `marked_stale`

### 7.4 是否复用 `memory_update_journal`

推荐方案：

1. `entity_state_current` 使用独立表
2. `entity_state_events` 首期不单独建新查询链路
3. 同时把实体状态变化映射为 `memory_updates`，`memory_layer = entity_state`
4. 继续落入现有 `memory_update_journal`

原因：

1. 复用现有审计 UI 和 API
2. 避免前端和 dashboard 重复建设
3. 首期更符合“最小侵入”

## 8. 实体状态 Patch 设计

### 8.1 为什么必须用 Patch

如果每轮都让模型输出“完整角色当前状态”，会有三个问题：

1. 容易因为幻觉重写旧事实
2. 很难审计到底发生了什么变化
3. rollback 时难以局部处理

因此建议每轮只输出 patch。

### 8.2 建议 Patch 结构

每轮抽取结果建议形如：

```json
{
  "turn": 12,
  "entities": [
    {
      "entity_id": "lorebook-char-zhangsan",
      "entity_type": "character",
      "display_name": "张三",
      "location": {
        "from": "钟楼外",
        "to": "钟楼顶层"
      },
      "inventory_add": ["铜钥匙"],
      "inventory_remove": [],
      "status_tags_add": ["受伤", "警觉"],
      "status_tags_remove": [],
      "companions_now": ["lorebook-char-lisi"],
      "short_goal": "寻找机关入口",
      "state_summary": "张三带伤登上钟楼顶层，手持铜钥匙，与李四同行。",
      "evidence": [
        "张三扶着伤口爬上钟楼顶层，手里攥着刚抢来的铜钥匙。"
      ],
      "confidence": 0.93
    }
  ]
}
```

### 8.3 合并规则

建议采用 deterministic merge：

1. `location.to` 明确时才更新 `current_location`
2. `inventory_add` 和 `inventory_remove` 做集合增减
3. `status_tags_add/remove` 做集合增减
4. `companions_now` 为显式覆盖字段，只在明确场景同伴变化时更新
5. `short_goal` 和 `state_summary` 允许覆盖
6. `evidence` 仅保留最近若干条

### 8.4 不确定信息处理

以下内容不直接写入 facts：

1. 模糊推断
2. 未明确说明的道具归属
3. 角色心态的过度解读
4. 同一回合存在明显冲突的信息

这些内容可以：

1. 进入 `metadata.hypotheses`
2. 进入 failed 或 warning event
3. 只保留在调试输出中

## 9. 后端执行流设计

### 9.1 新增生成后实体状态抽取步骤

建议在以下位置接入：

1. `StoryGenerator.generate_story(...)`
2. `StoryGenerator.generate_story_sync(...)`
3. `StoryGenerator.generate_story_stream(...)`

接入位置应与 `run_post_generation_updates(...)` 并列或被其统筹，而不是散落到 route 层。

### 9.2 建议新增服务层

建议新增一个独立服务，例如：

1. `EntityStateUpdateService`

职责：

1. 读取当前实体状态
2. 构造抽取输入
3. 执行结构化抽取
4. 做 schema 校验
5. 合并 patch
6. 写入 current 表
7. 生成 `entity_state` 类 memory update 事件

### 9.3 抽取输入建议

建议输入包括：

1. `user_input`
2. `generated_text`
3. 当前回合前的 `entity_state_current`
4. 可选 `runtime_state_snapshot`
5. 可选 `ScriptRoundContract`
6. principal character
7. active characters

其中，实体抽取范围应优先收敛到：

1. `principal_character_id`
2. `runtime_state.active_character_entry_ids`
3. 本轮显式选中的角色 lorebook entry

### 9.4 抽取失败降级

抽取失败不应阻断主故事生成。

降级行为建议为：

1. 主文本正常返回
2. 当前轮不更新 `entity_state_current`
3. 记录一条 `memory_layer = entity_state`、`status = failed` 的 journal 事件
4. 可附加 activation log 或 warning

## 10. 回滚、重生成与正文调整的重建方案

### 10.1 rollback

当前 `delete_last_message(...)` 已会：

1. 删除最后消息
2. 重建 cached context
3. 重建 history index
4. 重置 summary

建议新增：

1. 把相关实体状态标记为 `stale` 或直接 `reset`
2. 触发基于当前剩余消息的 entity rebuild
3. 把 rebuild 结果写入 `memory_updates`

### 10.2 regenerate

当前 regenerate 已先 reconcile 再重新生成。

建议新增：

1. reconcile 阶段同时处理 entity state stale/reset
2. 重新生成成功后，再进行新的 entity extraction
3. 使 `memory_updates` 中能看到：
   - `entity_state:reset` 或 `entity_state:marked_stale`
   - `entity_state:updated`

### 10.3 story adjustment commit

当前 `StoryConsistencyRebuildService` 已会：

1. 按 segment 重建 persisted messages
2. reset summary
3. rebuild history index

建议并列新增：

1. `rebuild_entity_state(...)`
2. 以 story segments 顺序重放生成 entity current state
3. 记录 `entity_state:rebuilt`

### 10.4 MVP 为什么优先用“全量重建”

原因：

1. 逻辑最稳
2. 便于对账
3. 与当前 message truth source 更一致
4. 适合先验证正确性

后续若性能成为问题，再考虑：

1. 按 `operation_id` 做增量回滚
2. 引入 checkpoint snapshot

## 11. API 契约设计

### 11.1 生成响应新增字段

建议在 `V2GenerateResponse` 和流式 done payload 中新增：

1. `entity_state_snapshot`
2. `entity_state_updates`

推荐含义：

- `entity_state_snapshot`：本轮生成后当前主要实体状态快照
- `entity_state_updates`：本轮产生的实体状态 patch 摘要

### 11.2 是否直接塞入现有 generate 响应

建议采用“两层策略”：

1. MVP 阶段：
   - 可放入 generate 响应
   - 只返回与当前会话和当前活跃角色相关的最小快照
2. 稳定阶段：
   - 增加独立查询接口
   - 避免长期膨胀 generate payload

### 11.3 独立查询接口建议

建议新增：

1. `GET /api/v2/story/session/{session_id}/entity-state`
   - 返回当前会话主要实体状态
2. `GET /api/v2/story/session/{session_id}/entity-state/events`
   - 返回实体状态时间线

如需与 story 绑定更强，也可提供：

1. `GET /api/v2/stories/{story_id}/entity-state`

### 11.4 与现有 memory routes 的关系

`memory_routes` 继续负责统一记忆时间线；`entity-state` 独立接口负责“当前实体状态读取”。

这样：

1. 时间线从 `memory_updates` 读取
2. 当前状态从 `entity_state_current` 读取

## 13. Prompt 注入策略

### 13.1 为什么不能全文注入所有实体状态

如果把完整 `entity_state_current` 全文注入 prompt，会带来：

1. token 膨胀
2. 噪声增加
3. 主次不分

### 13.2 建议注入方式

建议只注入“当前场景所需的最小实体摘要”：

1. 当前主角
2. principal character
3. active characters

摘要格式可形如：

```text
【当前实体状态】
- 张三：地点=钟楼顶层；状态=受伤、警觉；携带物=铜钥匙；同行=李四
- 李四：地点=钟楼顶层；状态=紧张；携带物=火折子
```

### 13.3 注入顺序

注入位置建议与 `summary_memory` 并列，而不是混进 `summary_text`。

## 16. 方案取舍与替代路径

### 16.1 为什么不直接增强 `summary_memory`

因为 `summary_memory` 的核心语义是压缩叙事记忆，不是结构化事实层。继续叠加会导致：

1. 读写职责混乱
2. rollback 语义不清
3. 可审计性变差

### 16.2 为什么不直接把动态状态写回 Lorebook

因为 Lorebook 更适合静态设定源，而不是回合级动态事实层。直接写回会导致：

1. 动静态语义混淆
2. 多故事共享同一世界书时互相污染
3. 回滚与重建成本升高

### 16.3 为什么不首期就做全实体图谱

因为当前项目最缺的是“可解释、可回滚、可展示的人物动态状态”，不是完整世界模型。先把人物实体追踪做成，再扩展到地点、物品、事件，更符合现有仓库的演进节奏。

---

## 17. 最终建议

本项目最合适的落地方式，不是继续强化“摘要记忆像不像实体状态”，而是明确补一层“实体状态真相源”。

建议采用以下最终方向：

1. 独立 `entity_state_current`
2. 复用 `memory_updates` 与 `memory_update_journal` 做变更审计
3. 在 `StoryGenerator` 生成后统一执行实体状态抽取
4. 在 `rollback / regenerate / adjustment commit` 时引入重建器
5. 前端先做只读面板与 diff 展示

该方向的优点是：

1. 与当前代码结构兼容
2. 对现有链路侵入可控
3. 可逐步灰度
4. 可审计、可回滚、可解释

从当前仓库现状看，这是最符合工程现实、产品价值和后续演进空间的方案。
