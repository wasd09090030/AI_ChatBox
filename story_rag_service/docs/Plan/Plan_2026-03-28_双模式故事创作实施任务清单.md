# Plan_2026-03-28_双模式故事创作实施任务清单

## 1. Planning Summary

### 1.1 目标

在现有《Plan_2026-03-28_双模式故事创作技术设计方案》的基础上，将双模式故事创作方案进一步拆解为按阶段执行的实现任务清单，用于指导后续真实开发。

本清单围绕两种模式展开：

1. 渐进式创作
2. 剧本主导创作

### 1.2 范围

本任务清单覆盖以下改动面：

1. 前端故事页与交互状态
2. 前端故事生成 API 契约
3. 后端故事请求模型
4. 严格模式运行态状态存储
5. 后端生成链路分叉
6. 严格模式下的过滤式 RAG
7. 严格模式一致性校验
8. 流式与非流式链路对齐
9. 兼容、迁移、联调和验收

### 1.3 约束

1. 必须基于现有仓库结构演进，不能绕开现有 StoryView、StoryGenerator、MemoryOrchestrator、retrieve_rag_context 再重做一套体系。
2. 渐进式创作必须尽量复用当前故事生成主链路。
3. 剧本主导创作必须引入独立的运行态真相源，而不是继续只依赖 `story metadata` 和前端传值。
4. 旧请求、旧故事、旧字段在兼容期必须仍能工作。

### 1.4 非范围

本清单不覆盖以下内容：

1. 剧本设计器 UI 重做
2. Lorebook / 世界书数据模型重构
3. 二次 LLM 审核型重一致性系统
4. 多用户协作与复杂时间线编辑器

## 2. Technical Direction

### 2.1 推荐实施方向

采用“兼容旧链路 + 增量引入新真相源 + 分阶段收紧严格模式”的路径。

总体策略：

1. 先建立双模式请求字段与兼容层。
2. 再建立 `ScriptRuntimeState`。
3. 先在非流式链路打通严格模式最小闭环。
4. 再收紧严格模式中的 RAG 和一致性校验。
5. 最后再做前端严格模式工作区与流式路径对齐。

### 2.2 现有模块复用原则

以下模块优先复用：

1. `frontend/src/views/StoryView.vue`
2. `frontend/src/domains/story/composables/useStoryGeneration.ts`
3. `frontend/src/domains/story/api/storyGenerationApi.ts`
4. `story_rag_service/models/story.py`
5. `story_rag_service/services/story_generator.py`
6. `story_rag_service/application/memory/orchestrator.py`
7. `story_rag_service/application/story_generation/retrieval.py`
8. `story_rag_service/models/script_design.py`
9. `story_rag_service/services/story_manager.py`

以下能力建议新增：

1. `ScriptRuntimeState` 模型与 repository
2. 严格模式 contract builder
3. 严格模式 consistency checker
4. 严格模式 runtime state API
5. 前端严格模式状态面板与运行态查询接口

### 2.3 关键取舍

#### 取舍 1：先非流式，后流式

原因：

1. 严格模式涉及 contract、校验、状态推进，非流式更易验证。
2. 流式路径可以在 done 事件后补状态推进，复杂度更高，应后置。

#### 取舍 2：先规则校验，后扩展高级校验

原因：

1. 首期目标是建立结构驱动闭环，而不是做复杂审查系统。
2. 规则校验已经足以覆盖阶段漂移、伏笔提前回收、未经授权实体等核心问题。

#### 取舍 3：StoryView 不拆页

原因：

1. 当前故事页已经承担故事库、输入、侧栏、控制面板、剧本选择等编排职责。
2. 新模式应作为该页面内的不同工作方式，而不是先拆出第二个完全独立页面。

## 3. 分阶段实施计划

1. `story_rag_service/models/story.py`
2. `frontend/src/domains/story/api/storyGenerationApi.ts`
3. `frontend/src/domains/story/composables/useStoryGeneration.ts`
4. `story_rag_service/api/v2/story/*`

### 主要任务

1. 定义 `creation_mode`
2. 定义 `progress_intent`
3. 定义 `runtime_state_id`
4. 设计旧字段 `follow_script_design` 的兼容期映射策略
5. 明确未传 `creation_mode` 时默认回落到渐进式

### 依赖

1. 无

### 验收

1. 新旧前端请求都能通过类型校验与路由接收。
2. 未传 `creation_mode` 的旧请求按原体验运行。

## Phase 2：ScriptRuntimeState 存储与接口

### 目标

1. 为严格剧本模式建立独立的运行态真相源。
2. 使用 SQLite 持久化运行态数据。
3. 提供运行态查询与更新基础接口。

### 主要改动面

1. `story_rag_service/services/database.py`
2. `story_rag_service/models/`
3. `story_rag_service/repositories/`
4. `story_rag_service/api/v2/story/*`
5. `story_rag_service/services/story_manager.py`

### 主要任务

1. 设计 `ScriptRuntimeState` 数据模型
2. 设计 SQLite 表结构
3. 新增 runtime repository
4. 新增查询、更新、初始化接口
5. 设计从旧 `story metadata` 初始化 runtime state 的兜底逻辑

### 依赖

1. Phase 1

### 验收

1. 可按 `story_id / session_id / script_design_id` 读取 runtime state。
2. 旧故事在没有 runtime state 时能生成初始运行态。
3. 文档明确 runtime state 与 `StoryContext`、`story metadata` 的职责边界。

### 风险控制

1. 兼容期允许 runtime state 和 metadata 共存，但必须写明真相源优先级。
2. 不在本阶段就把所有状态推进逻辑压进去。

## Phase 3：严格模式非流式最小闭环

### 目标

1. 在非流式路径打通严格模式的最小闭环。
2. 建立模式分发、contract 构建、结构驱动生成和状态推进的主路径。

### 主要改动面

1. `story_rag_service/api/v2/story/generation_routes.py`
2. `story_rag_service/graph/story_v2/nodes.py`
3. `story_rag_service/services/story_generator.py`
4. `story_rag_service/application/`

### 主要任务

1. 为 `creation_mode = scripted` 增加独立执行分支
2. 基于 `ScriptDesign + ScriptRuntimeState` 构建本轮 `round contract`
3. 解析当前 `stage / event`
4. 根据 `progress_intent` 设计本轮允许推进范围
5. 在生成后按规则决定是否推进 runtime state

### 依赖

1. Phase 2

### 验收

1. 非流式请求可显式进入严格模式。
2. 严格模式可读取当前 `stage / event` 并基于其生成。
3. 生成后可依据规则更新 runtime state。
4. 渐进式模式仍然复用现有主链路，不受影响。

### 风险控制

1. 严格模式只先做最小闭环，不在本阶段同时收紧 RAG 与前端 UI。
2. 优先复用现有 `StoryGenerator` 和 `MemoryOrchestrator`，避免新建平行生成服务。

## 5. Risk & Unknowns

### 风险 1：runtime state 与 metadata 双真相源冲突

影响：

1. 阶段与事件显示不一致
2. strict mode 推进错误

缓解：

1. 文档和实现中明确 strict mode 以 runtime state 为准
2. metadata 仅作为兼容层与初始化来源

### 风险 2：strict mode 过死

影响：

1. 生成内容变机械

缓解：

1. 仅把结构边界做硬约束
2. 保留 prompt 对本轮重点和表现形式的调节能力

### 风险 3：流式路径滞后

影响：

1. 用户在 strict mode 下会感觉流式与非流式行为不一致

缓解：

1. 先明确非流式为主验证路径
2. 流式在过渡期允许限制自动推进

### 风险 4：RAG scope 设计过宽或过窄

影响：

1. 过宽会带偏主线
2. 过窄会丢失必要背景知识

缓解：

1. 先使用 `linked_lorebook_entry_ids / participant_lorebook_entry_ids / linked_role_ids`
2. 保留少量兜底背景条目

### 风险 5：旧故事迁移不完整

影响：

1. 历史故事无法进入 strict mode 或状态异常

缓解：

1. 提供运行态初始化逻辑
2. 未初始化时回落为渐进式模式

## 6. Review Notes

1. 将前端严格模式工作区放到 runtime state 与非流式 strict mode 之后，避免先做 UI 再补后端真相源。

3. 将流式路径对齐后置，确保先在非流式路径建立可验证闭环。
4. 将迁移、联调与 smoke 收口为独立阶段，避免方案停留在设计层。
