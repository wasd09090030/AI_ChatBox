# Plan_2026-04-10_故事记忆系统合并重构方案

## 1. 背景与目标

当前仓库的故事主链路已经同时存在：

1. `summary_memory`
2. `runtime_state`
3. `entity_state_snapshot / entity_state_updates / world_update`
4. `memory_update_journal`

它们已经支撑起“可生成、可追踪、可重建”的基础能力，但产品层和系统层仍然是分裂的：

1. 用户看到的是多个概念：摘要、实体状态、世界更新、记忆事件
2. 后端写路径是混合的：`summary_memory_manager`、`story_runtime_manager`、`entity_state_manager`、`entity_patch_update_service` 并存
3. 前端记录模型也是分散的：`summaryMap`、`entityStateMap`、`entityUpdateMap`、`worldUpdateMap`、`runtimeMemoryEventsBySession`

基于当前仓库现状，本方案确定采用“**故事记忆系统合并**”方向：

1. 产品层 / 系统层统一叫 **故事记忆（Story Memory）**
2. **LLM patch 作为实体记忆主写路径**
3. 摘要记忆、实体记忆、运行态记忆统一收束到一套聚合契约
4. 前端同步重构为统一的故事记忆记录模型

### 1.1 本次方案要解决的问题

1. 当前“故事记忆”在产品层不是一个统一概念
2. 当前实体状态与摘要记忆、运行态记忆没有统一聚合视图
3. 当前前端存储模型分散，导致多个页面重复聚合
4. 当前实体 patch 已经成为主路径雏形，但尚未升格为系统层正式主路径
5. 当前旧实体状态功能仍混入主链路，职责边界不够清晰

### 1.2 本次设计目标

1. 将当前摘要 / 实体 / 运行态统一为 `Story Memory`
2. 建立统一响应字段 `story_memory`
3. 保持现有旧字段兼容，避免一次性大爆炸迁移
4. 让前端 `storySession` 内部出现统一的故事记忆记录模型
5. 逐步把旧 `entity_state_manager` 降级为 repair / rebuild / backfill 能力

### 1.3 非目标

1. 当前阶段不立刻删除旧实体状态功能
2. 当前阶段不强行合并数据库表
3. 当前阶段不要求所有 UI 只读新接口
4. 当前阶段不直接废弃 `memory_update_journal`

## 2. 现状基线与代码事实

### 2.1 后端服务容器已经是混合态

- `story_rag_service/api/service_context.py`

当前容器已同时装配：

1. `summary_memory_manager`
2. `story_runtime_manager`
3. `entity_state_manager`
4. `entity_patch_update_service`
5. `entity_state_event_repository`
6. `entity_state_event_replay_service`

这说明系统已经具备“旧实体状态 + 新 patch 事件流”双轨并存的事实基础。

### 2.2 Memory Orchestrator 尚未正式纳入实体记忆层

- `story_rag_service/application/memory/orchestrator.py`

当前 bundle 已有：

1. `episodic`
2. `semantic`
3. `profile`
4. `procedural`
5. `world`

但还没有正式的 `entity` 记忆层，因此故事记忆并未在建模层完成统一。

### 2.3 响应契约已经具备合并入口

- `story_rag_service/models/story.py`
- `story_rag_service/api/v2/schemas.py`

当前响应里已经同时包含：

1. `memory_updates`
2. `summary_memory_snapshot`
3. `runtime_state_snapshot`
4. `entity_state_snapshot`
5. `entity_state_updates`
6. `world_update`

这意味着可以在不破坏旧契约的前提下新增 `story_memory` 聚合字段。

### 2.4 前端记录模型仍是分裂状态

- `frontend/src/stores/storySession.ts`

当前 story session 相关记录拆分为：

1. `summaryMap`
2. `runtimeMemoryEventsBySession`
3. `entityStateMap`
4. `entityUpdateMap`
5. `worldUpdateMap`
6. `runtimeSummaryStateBySession`

这会导致故事页、dashboard 页各自做二次聚合。

### 2.5 前端产品命名也还未完全统一

- `frontend/src/components/story/StoryMemorySidebar.vue`
- `frontend/src/views/DashboardStoryMemoryView.vue`

虽然已经有 `StoryMemorySidebar` 命名，但实际文案和视图结构仍偏向“记忆变动 + patch 展示”的混合态，还不是统一的“故事记忆”概念。

## 3. 合并方向总览

### 3.1 产品层统一命名

统一使用：

1. **故事记忆总览**
2. **故事记忆时间线**
3. **实体记忆**
4. **语义记忆**
5. **运行态记忆**

不再在产品层把“世界更新”作为并列一级概念单独强调，而是把它收敛到故事记忆之下。

### 3.2 系统层统一模型

统一抽象为 `Story Memory`，内部拆为三层：

1. `semantic`
2. `entity`
3. `runtime`

同时保留：

4. `timeline`
5. `operation`

推荐结构：

```json
{
  "session_id": "story-xxx-v2",
  "story_id": "story_xxx",
  "world_id": "world_xxx",
  "operation": {
    "operation_id": "generate:uuid",
    "source": "generate",
    "status": "committed"
  },
  "semantic": {
    "summary_memory_snapshot": {}
  },
  "runtime": {
    "runtime_state_snapshot": {}
  },
  "entity": {
    "entity_state_snapshot": {},
    "entity_state_updates": [],
    "world_update": {}
  },
  "timeline": {
    "memory_updates": []
  }
}
```

### 3.3 写路径统一原则

1. `semantic` 继续由 `summary_memory_manager` 主写
2. `runtime` 继续由 `story_runtime_manager` 主写
3. `entity` 改为由 `entity_patch_update_service` 主写
4. `entity_state_manager` 降级为：
   - fallback rebuild
   - repair
   - backfill
   - 管理后台手动重建

## 4. 与现有方案相比的优势

### 4.1 概念统一

现有方案把摘要、实体状态、世界更新、记忆事件分拆展示；合并后，用户只需要理解“故事记忆”这一个概念。

### 4.2 主路径更清晰

现有方案中，旧 rebuild 与新 patch 共同参与主路径；合并后，实体记忆明确以 LLM patch 为主导，旧状态服务退到辅助层。

### 4.3 前端记录可复用

现有方案下，故事页和 dashboard 页都需要自行拼装 summary/entity/world/timeline；合并后统一通过 `storyMemorySessionMap` 或等价聚合模型读取。

### 4.4 更利于审计和回放

故事记忆通过 `operation_id` 串联当前轮次：

1. 文本生成
2. memory updates
3. entity patch
4. runtime transition
5. summary update

这使“本轮发生了什么”可以被完整解释。

### 4.5 更利于后期精简系统

第一阶段不删旧逻辑，但通过编排层和契约层合并，可以逐步把旧实体状态主功能安全退场。

## 5. 预期效果

### 5.1 用户层

1. 侧栏看到的是“故事记忆”而不是分散 patch 信息
2. dashboard 看到的是“故事记忆总览”而不是单纯记忆变动审计台
3. 每轮操作都能追到本轮的人物状态变化

### 5.2 系统层

1. 生成响应统一新增 `story_memory`
2. 流式 done payload 与非流式响应字段一致
3. 前端持久化模型出现统一故事记忆会话记录

### 5.3 后续阶段效果

在后续 Phase 2 ~ Phase 5 完成后，将进一步实现：

1. orchestrator 正式纳入 entity memory
2. UI 逐步切到统一 story memory 读模型
3. 旧实体状态主逻辑降级为 repair / rebuild

## 7. 风险与控制

### 7.1 patch 精度不足

风险：

1. 文本写了，但实体状态更新不完整

控制：

1. 保留 `entity_state_manager` fallback rebuild
2. 保留 `entity_state_event_replay_service`

### 7.2 前端兼容成本高

风险：

1. 现有故事页与 dashboard 广泛依赖旧 map

控制：

1. 第一阶段只新增聚合 map，不立刻删除旧 map
2. 先通过 `sync` 方式从旧 map 物化统一 story memory 记录

### 7.3 operation_id 对齐不稳

风险：

1. 一轮记忆更新无法稳定对账

控制：

1. 统一 story memory payload 从同一轮 `memory_updates / entity_state_updates` 推导 operation 元数据
