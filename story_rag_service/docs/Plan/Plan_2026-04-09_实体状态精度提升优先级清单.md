# Plan_2026-04-09_实体状态精度提升优先级清单

## 1. 背景

基于 `story_rag_service/docs/TestResult/Plan0409_DualRoutePatch_Validation_Run.json` 的最新复跑结果，当前“双路生成 + 结构化实体 patch”方案已经具备较强的工程可用性：

1. `generate / stream / regenerate` 三条主链路均可返回：
   - `entity_state_snapshot`
   - `entity_state_updates`
   - `world_update`
2. `operation_id / sequence` 对齐已经成立
3. rollback / rebuild / story segment rollback 已可优先基于事件回放恢复状态
4. `entity_state_events` 已成为稳定的字段级状态事件真相源

但同一份验证结果也表明，当前系统的主要短板已经从“链路是否打通”转移为“实体状态抽取精度是否稳定”。最新运行明细中唯一失败项是流式状态可见性断言，根因是地点短语 `地下仓库入口（向下石阶）` 没有被正确归一化，导致出现“位置未更新、物品已更新、目标仍沿用旧值”的半更新状态。

因此，本清单不再聚焦大链路搭建，而是聚焦“实体状态精度提升”的优先级排序、实施收益、预期效果与测试建议。

---

## 2. 当前问题归纳

### 2.1 已经做得好的地方

1. 显式字段已经可以稳定抽取：
   - `current_location`
   - `inventory`
   - `companions`
   - `short_goal`
2. patch 事件具备完整审计字段：
   - `event_id`
   - `operation_id`
   - `sequence`
   - `before / after`
   - `evidence_text`
3. patch 事件与 rebuild 事件已能统一进入 timeline，可用于回放、回滚、比对与审计

### 2.2 当前精度短板

1. 地点别名、过渡空间、修饰性地点短语归一化能力不足
2. patch 是逐字段独立应用，缺少实体级一致性约束，容易形成半更新状态
3. `short_goal`、`status_tags` 等弱结构字段容易滞后
4. 物品转移与同伴关系的双边一致性还不够强
5. 当前失败后的 repair 更偏整轮 fallback rebuild，缺少“单实体、单字段”粒度补救

---

## 3. 为什么要做这份优先级清单

### 3.1 优势

把实体状态精度问题拆成有优先级的清单，而不是继续做泛化重构，有以下优势：

1. **直接贴合现有仓库结构**
   - 主要修改点集中在：
     - `entity_patch_validator.py`
     - `entity_patch_applier.py`
     - `entity_patch_update_service.py`
   - 不需要重写 `StoryGenerator` 主链路
2. **优先修复真实暴露的问题**
   - 不是“猜测式优化”，而是围绕最新 smoke 的失败项与 warning 定向修复
3. **对现有前后端契约冲击小**
   - 优先在后端归一化、repair、warning 与 metadata 层增强
   - 可避免频繁改动前端 schema
4. **收益可快速量化**
   - 每一批优化都可以通过 smoke 用例和回归断言验证
5. **适合渐进式演进**
   - 先把高频、高收益问题做稳，再继续做复杂关系场景

### 3.2 期望达到的效果

如果按本清单推进，预期可以达到以下效果：

1. 流式和非流式链路的实体状态结果更一致
2. “位置旧、物品新、目标旧”这类半更新状态显著减少
3. 地点抽取对“入口 / 门口 / 平台边缘 / 向下石阶”类短语更稳
4. rollback / rebuild 后的状态与 patch 事件更容易对账
5. 前端看到的 patch 时间线与最终实体快照更一致
6. 实体状态系统从“可用”进一步提升到“更可信、更可依赖”

---

## 4. 优先级清单

## P0：必须优先完成

### P0-1 地点别名与模糊地点归一化

#### 问题

当前地点校验对精确命中依赖较强，遇到“地下仓库入口（向下石阶）”“钟楼顶层平台边缘”这类叙事性地点短语时，容易判为未知地点。

#### 原因

1. 当前 `location_lookup` 更偏标准名称匹配
2. 缺少括号内容剥离、方位修饰词裁剪、入口/通道等别名映射

#### 可能造成的影响

1. `current_location` 无法更新
2. 物品、目标、同行等字段仍可能更新，形成半更新状态
3. 流式场景更容易出现 patch 数量不足或状态漂移

#### 建议实施

1. 在 `entity_patch_validator.py` 中新增地点标准化步骤：
   - 去括号
   - 去修饰语
   - alias 映射
2. 支持“子地点 -> 主地点”的归一化：
   - `地下仓库入口（向下石阶）` → `地下仓库`
   - `钟楼顶层平台边缘` → `钟楼顶层`
3. 保留原始短语到 metadata，避免信息完全丢失

#### 预期效果

1. `current_location` 的成功更新率明显提升
2. 流式状态可见性用例转绿
3. 地点字段的 warning 数量下降

### P0-2 半更新保护

#### 问题

目前 patch 主要是逐字段独立校验和应用，缺少“同一实体本轮更新是否整体一致”的约束。

#### 原因

1. patch validator 更偏字段合法性
2. applier 更偏事件落地与快照投影
3. 还没有实体级一致性规则

#### 可能造成的影响

1. 位置未更新但物品已移除
2. 目标仍保留旧值
3. 当前快照看似合法，但叙事语义已经不一致

#### 建议实施

1. 对同一实体本轮 patch 做一致性检查：
   - 如果存在明显移动语义，但地点字段失败，记录强 warning
   - 如果存在物品转移、目标切换，但地点仍旧值，触发 repair 候选
2. 将“一致性风险”体现在 `world_update.entity_patch.warnings`

#### 预期效果

1. 半更新状态减少
2. 当前状态与 patch 时间线更一致
3. 测试失败时更容易定位问题根因

### P0-3 单实体 repair 机制

#### 问题

现在补救主要偏整轮 fallback rebuild，粒度太大。

#### 原因

1. 当前 fallback 更偏“整个 session / story 重建”
2. 缺少对“已识别实体，但关键字段缺失”的局部补偿

#### 可能造成的影响

1. patch 主路径明明大部分成功，仍需依赖较重的 rebuild
2. repair 成本高，语义不够精确

#### 建议实施

1. 新增单实体 repair 能力：
   - 输入：该实体当前快照 + 本轮相关文本 + 已提取 patch
   - 输出：仅补关键缺失字段，如 `current_location` / `short_goal`
2. 仅在关键字段失败时触发，避免增加全局成本

#### 预期效果

1. patch 主路径命中率提升
2. 更少依赖整轮 rebuild
3. 流式和非流式链路稳定性更高

---

## P1：高价值增强

### P1-1 `short_goal` 提取增强

#### 问题

`short_goal` 往往写在对白、行动意图或下一步计划里，当前容易滞后。

#### 可能造成的影响

1. 目标字段保留旧值
2. 当前快照对“接下来要做什么”的刻画不够可信

#### 建议实施

1. 在 `entity_patch_prompt.py` 中强化 few-shot 或说明：
   - “先去……”
   - “下一步……”
   - “目标是……”
2. 在 validator 中增加“目标与地点/动作冲突时降权或清空”的规则

#### 预期效果

1. `short_goal` 与当前轮叙事更一致
2. rollback 后目标字段回退更可靠

### P1-2 物品转移双边一致性

#### 问题

“A 把钥匙递给 B”这类场景容易只更新一边。

#### 可能造成的影响

1. A 失去物品，但 B 没拿到
2. 物品在系统里“消失”

#### 建议实施

1. 增加双边转移规则：
   - `A.inventory remove x`
   - `B.inventory add x`
2. 若只识别到一边，记 warning

#### 预期效果

1. 物品转移类状态更可信
2. inventory 字段在多人场景更稳定

### P1-3 `companions` 增删闭环

#### 问题

当前同行关系更容易 add，不够擅长 remove。

#### 可能造成的影响

1. 分头行动后仍被记录为同行
2. 同伴关系随故事推进逐渐失真

#### 建议实施

1. 增强：
   - 分头行动
   - 独自进入
   - 留在原地
   - 失散
   的 `companions remove`
2. 保证双边一致

#### 预期效果

1. 同伴关系不再只增不减
2. 人物态势板更符合实际剧情

---

## P2：持续优化项

### P2-1 `status_tags` 高精度规则增强

#### 目标

提高“警觉 / 受伤 / 疲惫 / 昏迷”等高价值状态标签的命中率。

#### 建议实施

1. 仅做少量高 precision 标签
2. 通过 validator 做弱规则加权，而不是一次性引入大量状态枚举

#### 预期效果

1. `status_tags` 更有内容
2. 误报可控

### P2-2 置信度驱动策略

#### 目标

让 `confidence` 不只是记录字段，而能真正驱动“应用 / warning / repair”策略。

#### 预期效果

1. 低置信 patch 不轻易污染快照
2. repair 触发更可控

### P2-3 前端可观测性增强

#### 目标

让前端更直观显示：

1. fallback 是否发生
2. warning 数量
3. 是否触发 repair
4. 当前 patch 置信度是否偏低

#### 预期效果

1. 调试成本下降
2. 产品侧更容易理解当前状态可信度

## 5. 建议测试用例

以下测试建议应优先加入现有 smoke / validation 脚本，或扩展为更细的回归集。

### 5.1 地点归一化类

1. `张三来到地下仓库入口（向下石阶）`
   - 预期：`current_location = 地下仓库` 或 `地下仓库入口`
2. `李四站在钟楼顶层平台边缘`
   - 预期：`current_location = 钟楼顶层`
3. `两人停在仓库门口观察`
   - 预期：归一化到已知地点，不得直接跳过

### 5.2 半更新保护类

1. `张三把铜钥匙放回原处，然后转身下到地下仓库`
   - 预期：
     - `inventory remove 铜钥匙`
     - `current_location = 地下仓库`
     - 若地点失败，必须有 warning / repair 痕迹
2. `李四改口说先封锁入口，不去档案馆`
   - 预期：
     - `short_goal` 更新
     - 旧目标不得无痕保留

### 5.3 物品转移类

1. `张三把铜钥匙递给李四`
   - 预期：
     - 张三 `inventory remove 铜钥匙`
     - 李四 `inventory add 铜钥匙`
2. `李四把火折子丢在仓库门口`
   - 预期：
     - 李四 `inventory remove 火折子`
     - 若未来支持世界物件层，可同时产生 world update

### 5.4 同伴关系类

1. `两人分头行动，张三留在钟楼，李四独自下仓库`
   - 预期：
     - 双边 `companions remove`
     - 地点分别更新
2. `张三又追上李四，两人重新并肩前进`
   - 预期：
     - 双边 `companions add`

### 5.5 状态标签类

1. `张三拔刀警戒，压低声音示意李四停下`
   - 预期：`status_tags add 警觉`
2. `李四肩头中刀，踉跄后退`
   - 预期：`status_tags add 受伤`

### 5.6 repair 与回放类

1. 制造地点失败但其他字段成功的场景
   - 预期：触发单实体 repair，而不是无感半更新
2. rollback 后检查 repair 产物是否与 event replay 对齐
3. regenerate 后检查新 patch 是否正确覆盖旧 patch 影响

## 6. 推荐实施顺序

### 第一批：必须优先完成

1. 地点别名与模糊地点归一化
2. 半更新保护
3. 单实体 repair

### 第二批：高价值增强

1. `short_goal` 提取增强
2. 物品转移双边一致性
3. `companions` 增删闭环

### 第三批：优化项

1. `status_tags` 高精度规则增强
2. `confidence` 驱动策略
3. 前端可观测性增强

---

## 7. 结论

当前双路生成已经进入“工程可用”阶段，但实体状态系统若要进一步从“可追踪”提升到“高可信”，最值得优先投入的不是大规模重构，而是：

1. 地点归一化
2. 半更新保护
3. 单实体 repair

这三项投入最小、收益最高、对现有架构冲击最小，并且能够直接解决最新验证结果中暴露出的真实问题。
